# Requirements: EMEL v1.21 Quality Gate Selective Runner Optimization

**Defined:** 2026-05-02
**Source:** GitHub issue #58
**Core Value:** Prove real end-to-end behavior with explicit SML orchestration and
parity-oriented verification before widening API surface or model scope.

## v1 Requirements

### Gate Contract

- [x] **GATE-01**: Maintainer can continue to use `scripts/quality_gates.sh` as the mandatory
  top-level gate for implementation changes.
- [x] **GATE-02**: Maintainer can see, in the quality-gate output, which major lanes ran, which
  parity or benchmark runners were selected or skipped, why that choice was made, and how long
  each lane took.
- [x] **GATE-03**: Maintainer can rely on the gate to preserve required coverage, lint snapshot,
  docs, fuzz smoke, parity, benchmark, and lane-isolation enforcement unless a lane is irrelevant
  to the scoped changed files by existing policy.

### Impact Resolution

- [x] **IMPACT-01**: Maintainer can derive the impacted parity runner set from
  `parity_dependency_manifest/v1` data when the manifest state is present, fresh, and
  trustworthy.
- [x] **IMPACT-02**: Maintainer can derive the impacted benchmark runner set from
  `bench_dependency_manifest/v1` data when the manifest state is present, fresh, and trustworthy.
- [x] **IMPACT-03**: Maintainer gets a conservative full relevant parity or benchmark runner set
  when manifest data is missing, stale, uncertain, malformed, or impact resolution fails.

### Selective Runner Execution

- [x] **RUNNER-01**: Maintainer can run only the impacted parity runners through the maintained
  paritychecker entrypoint when impact resolution is trustworthy.
- [x] **RUNNER-02**: Maintainer can run only the impacted benchmark runners through the maintained
  benchmark entrypoint when impact resolution is trustworthy.
- [x] **RUNNER-03**: Maintainer can reproduce selective-runner decisions from checked-in manifest
  baselines and changed-file inputs without relying on tool-only scaffolds or hidden state.

### Parallel Gate Orchestration

- [x] **PAR-01**: Maintainer can run independent quality-gate lanes in parallel where their build,
  test, runtime, and snapshot artifacts do not conflict.
- [x] **PAR-02**: Maintainer gets deterministic enough aggregated logs and exit status reporting to
  debug local and CI failures after parallel gate execution.

### Validation And Performance

- [x] **VAL-01**: Maintainer has focused regression coverage for impact resolution, conservative
  fallback, selective parity execution, selective benchmark execution, and parallel result
  reporting.
- [x] **VAL-02**: Maintainer has end-to-end quality-gate evidence showing materially improved wall
  time for representative localized changes while preserving required validation lanes.
- [x] **VAL-03**: Maintainer can close the milestone only after source-backed audit evidence proves
  all active quality-gate optimization requirements are satisfied.

## v2 Requirements

Deferred to future releases. Tracked but not in current roadmap.

### Gate Expansion

- **GATE-EXT-01**: Maintainer can extend manifest-backed impact selection to additional future
  tool families after those tools publish trustworthy dependency manifests.
- **GATE-EXT-02**: Maintainer can add CI-native distributed execution once the local gate
  orchestration is deterministic and source-backed.
- **GATE-EXT-03**: Maintainer can publish richer historical timing dashboards after the required
  local/CI gate output remains clear and stable.

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Skipping `scripts/quality_gates.sh` | Issue #58 explicitly keeps the top-level gate mandatory. |
| Permissive manifest fallback | Missing, stale, uncertain, or failed impact resolution must run the affected full relevant runner set. |
| Hash-only quality-gate skipping | Existing project rules require conservative build-graph dependency manifests, not hash-only trust. |
| Changing parity or benchmark semantics | v1.21 optimizes orchestration, not maintained compare or benchmark meaning. |
| Weakening coverage, lint, docs, fuzz, parity, or benchmark thresholds | The milestone goal is less unnecessary work, not lower confidence. |
| Updating snapshots solely to show speed improvement | Snapshot changes require their normal source-backed justification and user approval. |
| Public third-party runner SDK | The work stays inside repo-owned quality-gate, paritychecker, and benchmark tooling. |
