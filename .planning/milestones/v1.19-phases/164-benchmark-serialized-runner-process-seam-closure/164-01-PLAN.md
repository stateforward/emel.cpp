# Plan 164-01: Benchmark Serialized Runner Process Seam Closure

<rule_constraints>
- Keep the normal `bench_runner` invocation path behaviorally stable.
- Add the live process seam through the runner boundary, not as a test-only helper.
- Use focused tests that invoke the built `bench_runner` process and parse the serialized result.
</rule_constraints>

## Goal

Close `RUNNER-02` by wiring `bench_runner_request/v1` and `bench_runner_result/v1` into a live
benchmark runner process entrypoint.

## Tasks

1. Add exclusive serialized request/result process flags to `run_bench_cli(...)`.
2. Read and parse the request payload, validate suite/mode/jsonl constraints, execute the selected
   registered runner path, and write a serialized result payload.
3. Refactor runner dispatch just enough to share execution between normal CLI and serialized
   process mode without changing ordinary output behavior.
4. Add `bench_runner_tests` cases for a successful live process request, malformed request failure,
   and unknown suite failure.
5. Build and run the focused benchmark runner tests plus scoped quality gates.

## Completion Notes

- Added exclusive `--run-serialized-request <path>` and `--write-serialized-result <path>` process
  flags to the production `bench_runner` CLI.
- Shared normal CLI and serialized-process dispatch through `execute_runner_request(...)`.
- Added pre-dispatch validation so malformed payloads, unknown modes, unknown suites, compiled
  suite mismatches, and conflicting JSONL modes write deterministic `bench_runner_result/v1`
  errors.
- Added live binary tests for success and fail-closed process-seam behavior.
