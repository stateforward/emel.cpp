# v1.15 Milestone Audit: ARM Sortformer Diarization GGUF Slice

## Outcome

Status: `passed`

`v1.15` is ready for milestone closeout. The live source-backed path now proves the maintained ARM
Sortformer diarization slice end to end: pinned GGUF + pinned AMI WAV fixture, EMEL-owned loader and
runtime, PyTorch/NeMo parity reference, ONNX Runtime CPU single-thread benchmark reference, and
generated benchmark/docs publication.

The prior audit blocker is closed. EMEL, PyTorch/NeMo, and ONNX now agree at `output_dim=17` and
checksum `4249677247906920305`; EMEL also beats the ONNX CPU single-thread benchmark reference in
the strict Phase `93` record set.

## Source-Backed Claim Trace

The closeout claim was verified from source paths and generated records, not planning artifacts
alone:

- `tools/bench/diarization/sortformer_fixture.hpp` loads
  `tests/models/diar_streaming_sortformer_4spk-v2.1.gguf`, prepares
  `tests/fixtures/diarization/ami_en2002b_mix_headset_137.00_152.04_16khz_mono.wav`, builds the
  Sortformer execution contract, and drives `emel::diarization::sortformer::pipeline::sm`.
- `src/emel/diarization/sortformer/pipeline/sm.hpp` orchestrates request validation, acoustic
  feature extraction, encoder execution, executor dispatch, speaker probabilities, and segment
  decoding through explicit SML transitions.
- `src/emel/diarization/sortformer/encoder/detail.cpp` and
  `src/emel/diarization/sortformer/transformer/detail.cpp` keep dense/matmul work routed through
  Sortformer runtime/kernel-owned helpers rather than benchmark-only compute.
- `tools/bench/diarization_compare.py` emits separate EMEL, recorded regression, ONNX, and
  PyTorch/NeMo records under the shared `diarization_compare/v1` contract.
- `tools/bench/diarization_sortformer_pytorch_reference.py` uses the official
  `SortformerEncLabelModel.diarize(..., include_tensor_outputs=True)` model-card path and labels it
  `reference_role=parity_reference`.
- `tools/bench/diarization_sortformer_onnx_reference.py` uses ONNX Runtime with
  `CPUExecutionProvider`, `intra_op=1`, `inter_op=1`, and `ORT_SEQUENTIAL`; it labels the lane
  `reference_role=benchmark_reference` and records the actual providers.
- `build/diarization_compare_phase93_score_prenorm_restored_pytorch/compare_summary.json` reports
  `failed=false` and exact match for EMEL against both executable reference lanes.
- `snapshots/bench/benchmarks_compare.txt` and `docs/benchmarks.md` publish the maintained
  `17`-segment checksum and distinguish PyTorch parity from ONNX benchmark reference.
