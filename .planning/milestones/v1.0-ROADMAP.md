# Milestone v1.0: EMEL Llama-68M Generation Slice

**Status:** ✅ SHIPPED 2026-03-08
**Phases:** 1-6
**Total Plans:** 15

## Overview

v1.0 proved one narrow brownfield generation slice inside `tools/paritychecker/`: the repo can run
the canonical `tests/models/Llama-68M-Chat-v1-Q2_K.gguf` fixture through the real EMEL GGUF/model
loading path, initialize the existing generator actor graph, execute one bounded deterministic
generation request, compare the result against the `llama.cpp` reference path, and hold the whole
flow behind normal subprocess and repo-gate coverage.

## Milestone Summary

**Decimal Phases:**

- Phase 1.1: implement the gguf::loader that's just stubbed (inserted after Phase 1 to unblock the
  real EMEL load path)

**Key Decisions:**

- Target `tests/models/Llama-68M-Chat-v1-Q2_K.gguf` first to keep the slice narrow and local.
- Define done at the paritychecker boundary instead of widening the public C API.
- Insert Phase 1.1 rather than carrying the stubbed GGUF loader deeper into the milestone.
- Close the late `HARN-02` audit gap with canonical-path enforcement instead of weakening the
  requirement wording.
- Reuse the existing paritychecker and quality-gate scripts as the release proof surface.

**Issues Resolved:**

- The stubbed GGUF loader no longer blocks end-to-end model loading.
- Generation mode no longer stops at load or initialize seams; it reaches bounded parity-checked
  generation.
- The first-slice fixture contract is now canonical-path based rather than basename-only.

**Issues Deferred:**

- The paritychecker loader-to-generator handoff still backfills some runtime data from `llama.cpp`
  before initialize.

**Technical Debt Incurred:**

- `scripts/quality_gates.sh` still treats benchmark snapshot drift as non-blocking repo policy.
