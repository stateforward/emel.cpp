# Phase 96 Plan 01: Native Quant Variant Kernels (FULL NEON)

**Phase Goal:** Add native kernel-level support for the narrowed maintained Whisper quant
variant family `{q8_0, q4_0, q4_1}` (per Phase 95 narrowing approved 2026-04-25, agent-0
message `J2bGANuAk61CW6P6`), including ARM optimized dispatch proof.

**Plan Goal:** Land real ARM NEON intrinsics for `dot_q4_0_q8_0` and `dot_q4_1_q8_0` with
explicit SML routing through `simd_op_mul_mat_q4_0_vector` and `simd_op_mul_mat_q4_1_vector`
guard/action pairs, keep the q5_0/q8_0 paths intact, prove numeric parity vs scalar, and
prove SML dispatch counters increment exactly once for routed events.

## Tasks

### Task 1 — Add `quantize_row_q4_0_ref` And `quantize_row_q4_1_ref`

- Add two `quantize_row_q*_ref` helpers in `src/emel/kernel/detail.hpp` immediately above
  `dequantize_row_q4_0`/`dequantize_row_q4_1`. Mirror the existing
  `quantize_row_q5_0_ref` semantics: scan the dense row in 32-element groups, find the
  per-block max-absolute (q4_0) or min/max range (q4_1), pick `d` (and `m` for q4_1) so
  the recovered quantized value lands on the correct domain, pack two values per byte.
- These helpers are kernel-owned data-plane code, not orchestration; they have no
  branching beyond bounded data-plane work and do not select between behaviors.
- Acceptance: round-tripping dense rows via `quantize_row_q4_*_ref` then
  `dequantize_row_q4_*` recovers values within an epsilon proportional to the q4 step.

### Task 3 — Add Real NEON Intrinsics For `dot_q4_0_q8_0`

In `src/emel/kernel/aarch64/actions.hpp`, just below the q5_0 NEON helpers:

- `dot_q4_0_q8_0_row_neon(lhs, rhs, block_count)` — for each block:
  - Load `qs[16]` as `uint8x16_t`. Mask low nibble (`vandq_u8(..., 0x0F)`) and shift to
    high nibble (`vshrq_n_u8(..., 4)`). Subtract 8 to recover signed values
    (`vsubq_s8(vreinterpretq_s8_u8(...), vdupq_n_s8(8))`).
  - Load rhs `qs` halves as two `int8x16_t`.
  - DOTPROD path: two `vdotq_s32(acc, lhs_half, rhs_half)`, then horizontal sum.
  - Vanilla NEON fallback: `vmull_s8` + `vpaddlq_s16` + `vaddq_s32`, mirroring the q5_0
    fallback pattern.
  - Multiply by `lhs.d * rhs.d` (fp16→fp32) and accumulate into `sumf`.
- `dot_q4_0_q8_0_4rows_neon(row0..row3, rhs, block_count, out[4])` — same body but
  loads four LHS rows per block and accumulates four sums in parallel.
- Both functions return scalar via `dot_q4_0_q8_0_row_scalar` when `__aarch64__` /
  `__ARM_NEON` is unavailable.
- Acceptance: numeric parity vs scalar within `1.0e-5f` for randomized blocks.

### Task 4 — Add Real NEON Intrinsics For `dot_q4_1_q8_0`

Same pattern but the q4_1 contract is `x = q * d + m`:

- For each block, compute `int_dot = sum(q_unsigned * rhs.qs)` and
  `rhs_sum = sum(rhs.qs)`.
- `int_dot` follows the same NEON shape as q4_0 (no `-8`, but the lhs lanes are unsigned;
  reinterpret as `int8x16_t` is fine because all values are in `[0, 15]`, well below 127).
- For `rhs_sum`, accumulate via `vaddlvq_s8` (or pairwise widen + horizontal sum on
  fallback path). Compute scaled result as
  `lhs.d * int_dot * rhs.d + lhs.m * rhs.d * rhs_sum`.
- 4-row variant accumulates four lhs rows + one shared rhs sum per block.
- Acceptance: numeric parity vs scalar within `1.0e-5f` for randomized blocks.

### Task 5 — Add Support, Acceptance, And Execute Functions

In `src/emel/kernel/aarch64/actions.hpp`, mirror the q5_0 vector pattern:

- `neon_q4_0_vector_supported()` — true on `aarch64 && __ARM_NEON && __ARM_FEATURE_DOTPROD`,
  else false (matching q5_0 surface; the widening fallback is still used internally when
  DOTPROD is missing, but the SML route is gated to the well-tested DOTPROD configuration).

  with `dtype_q4_0` and `QK4_0`.
- `can_use_neon_mul_mat_q4_0_vector(request, neon_available)` — `neon_available &&
  neon_q4_0_vector_supported() && can_run_neon_mul_mat_q4_0_vector_request(request)`.
- `execute_neon_mul_mat_q4_0_vector_unchecked(request)` — same body as q5_0 vector exec,
  switching the LHS block type to `block_q4_0`, the dot helpers to the q4_0 NEON pair,
  and the input quantization to `quantize_row_q8_0_strided` (unchanged).
- `execute_neon_mul_mat_q4_0_vector(request)` — same wrapper.
- Repeat for q4_1.
- Acceptance: build clean; both per-row and 4-row code paths exercised.

## Risk / Notes

- The q4_1 NEON path is more complex than q4_0 because of the `lhs.m * rhs.d * rhs_sum`
  term. Both DOTPROD and widening fallbacks must compute the rhs sum correctly. The
  fallback path uses `vaddlvq_s8` (DOTPROD-required) or pairwise widen + horizontal sum
  for vanilla NEON; double-check the signed accumulation width.
- The `simd_op_mul_mat_generic` fall-through guard list MUST be updated when new
  guards are added; otherwise the generic backend takes over and dispatch counters do
  not increment. This is the most likely source of subtle test failure.
- AGENTS.md routing rule: the new guards live in `guards.hpp`, the new arithmetic in
  `actions.hpp` (kernel layer), and the routing decisions are encoded in `sm.hpp`
  transitions. No runtime branching is added in `actions.hpp` outside of compile-time
  `#if` for the DOTPROD fallback.
- The current build host is macOS arm64 with `-mcpu=native+dotprod+i8mm` per
  `CMakeLists.txt`. DOTPROD path will be exercised at runtime there.
