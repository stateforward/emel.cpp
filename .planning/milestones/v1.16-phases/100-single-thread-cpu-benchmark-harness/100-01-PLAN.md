# Phase 100 Plan 01: Single-Thread CPU Benchmark Harness

<rule_constraints>
## Benchmark Truth

- Run the same pinned model/audio fixture pair as Phase 99.
- Invoke the reference lane with one thread, one processor, and GPU disabled.
- Record warmup/iteration counts, host identity, backend identity, model/audio checksums, and
  timing fields.

## Isolation

- Keep `whisper.cpp` as an external process under the reference lane.
- Keep EMEL benchmark output driven through EMEL-owned loader/runtime actors.
</rule_constraints>

## Tasks

1. Extend EMEL runner timing output.
   - Add model/audio load, binding, contract, encode, decode, publish, and total timing fields.
   - Mark the runner output as single-thread CPU metadata.

2. Add benchmark driver.
   - Add a Python benchmark harness that runs warmups and measured iterations for both lanes.
   - Parse `whisper.cpp` timing output and collect EMEL runner timing fields.
   - Write raw `whisper_benchmark/v1` JSONL records and a summary.

3. Add operator wrapper and gate.
   - Add `scripts/bench_whisper_single_thread.sh`.
   - Add `whisper_single_thread` as a custom quality-gate benchmark suite.

4. Verify and record.
   - Run the wrapper.
   - Run the scoped quality gate.
   - Store Phase 100 completion artifacts with benchmark evidence paths.

## Acceptance Criteria

- Both lanes run with one thread/processor and CPU-only reference flags.
- Benchmark records include warmup count, iteration count, wall time, stage timings where
  available, host identity, backend version/build identity, model checksum, and audio checksum.
- The scoped quality gate passes through `EMEL_QUALITY_GATES_BENCH_SUITE=whisper_single_thread`.
