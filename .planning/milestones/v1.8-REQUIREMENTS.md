# Requirements: EMEL

**Defined:** 2026-04-02
**Core Value:** Prove real end-to-end behavior with explicit SML orchestration and parity-oriented
verification before widening API surface or model scope.

## v1 Requirements

### Workload Contract

- [x] **WORK-01**: The embedded-size workflow fixes one maintained workload boundary at
  `tests/models/Qwen3-0.6B-Q8_0.gguf` with the structured `hello` prompt and `max_tokens=1`.
- [x] **WORK-02**: The published measurement stays on final linked executables and does not fall
  back to library artifact size as the primary repo claim.

### EMEL E2E Probe

- [x] **E2E-01**: EMEL builds a final linked Qwen3 runner executable that loads model state,
  tokenization/conditioning state, and generates the first token on the maintained workload.
- [x] **E2E-02**: The published EMEL runner remains EMEL-owned end to end and does not bootstrap
  vocab or generation behavior through `llama.cpp` or `ggml`.

### Reference Comparators

- [x] **REF-01**: `llama.cpp` builds a final linked reference runner executable for the same
  fixture, prompt contract, and bounded generation slice.

### Smoke And Publication

- [x] **SMOKE-01**: When the maintained Qwen3 fixture is present locally, the size workflow
  smoke-runs every published executable row on the same `hello` -> first-token path and records
  the result.
- [x] **PUB-01**: `scripts/embedded_size.sh` emits raw, stripped, and section byte measurements
  plus exact build/workload metadata for every published executable row.
- [x] **PUB-02**: Generated README docs publish the embedded-size snapshot with runtime-smoke
  status and explicit scope/caveat wording for the maintained Qwen3 slice.

## v2 Requirements

### Broader Measurement Coverage

- **BUNDLE-01**: Publish deployable bundle-size accounting that includes required model and
  ancillary assets after executable-only measurement is stable.
- **MODEL-03**: Broaden beyond the canonical `Qwen3-0.6B-Q8_0.gguf` maintained slice only after
  the first executable-size claim is shipped and trusted.

### Performance And Policy

- **PERF-01**: Optimize EMEL executable size after the truthful baseline is published and
  attributable.
- **GATE-01**: Decide later whether executable-size regressions should become a blocking quality
  gate once the signal is stable enough to justify it.
- **LIQ-01**: Resume the deferred Liquid milestone after v1.8 closes.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Whole-product feature-parity size claims | v1.8 only proves one matched executable boundary, not total shipped product equivalence. |
| Library artifact size as the published truth surface | The user explicitly redirected the work toward real executable measurement for embedded relevance. |
| Deployable bundle size in v1.8 | Binary size and asset/package size are different measurements and should not be blurred. |
| Additional comparator runtimes or alternate model formats | They would widen the apples-to-apples Qwen3 executable claim beyond the milestone boundary. |
| Broader Liquid-family runtime work | The previously pending Liquid plan is deferred until after v1.8. |
| Broader Qwen-family or multi-fixture support | Keep the milestone on one canonical maintained Qwen3 slice. |
| Blocking quality-gate enforcement for size regressions | The size surface is still new and should remain reference-only until proven stable. |
