# Milestone v1.1 Requirements: EMEL Llama-68M Generation Benchmark

**Status:** ✅ SHIPPED 2026-03-11
**Audit:** passed
**Source milestone:** v1.1

## Final Outcome

v1.1 shipped all 10 planned requirements for the truthful Llama-68M benchmark slice. The milestone
added one canonical generation benchmark in `tools/bench`, replaced the circular reference-backed
decode seam with a shared native EMEL backend, published stable EMEL-vs-`llama.cpp` compare
evidence, and integrated the benchmark into the existing snapshot plus docs-generation workflow.

## v1 Requirements

### Benchmark Harness

- [x] **BENCH-01**: User can run one `tools/bench` generation benchmark for
  `tests/models/Llama-68M-Chat-v1-Q2_K.gguf`.
- [x] **BENCH-02**: User can benchmark a deterministic bounded generation workload with explicit
  prompt and token-budget settings.
- [x] **BENCH-03**: User can tell from benchmark output what portion of the generation path is
  being measured.

### Comparison

- [x] **TRUTH-01**: User can trust that the EMEL generation path does not call
  `llama_decode`/`llama_get_logits_ith` for its own decode or logits extraction.
- [x] **TRUTH-02**: User can run the canonical paritychecker generation slice through a native
  EMEL decode backend while keeping the explicit `llama.cpp` reference path separate.
- [x] **TRUTH-03**: User can run the canonical `tools/bench` EMEL generation case through the same
  native EMEL decode backend used by paritychecker.
- [x] **COMP-01**: User can run EMEL and `llama.cpp` generation benchmarks for the same canonical
  Llama-68M workload through the existing compare flow.
- [x] **COMP-02**: User can inspect per-case benchmark output showing EMEL timing, `llama.cpp`
  timing, and the ratio between them.

### Verification

- [x] **VBEN-01**: User can run the generation benchmark through the existing `scripts/bench.sh`
  compare path without creating a second benchmark workflow.
- [x] **VBEN-02**: User can maintain benchmark snapshots for the new generation case through the
  existing snapshot/update tooling.

## Outcome Notes

- Phase 07.1 was inserted after Phase 7 when the initial benchmark path was shown to be circular.
  That corrective phase is what made the published EMEL benchmark evidence truthful.
- The benchmark milestone stayed within the agreed acceptance boundary: `tools/bench`,
  `tools/paritychecker`, and the existing snapshot/docs workflow only. It did not widen into a
  new public API or a multi-model benchmark matrix.
- The audit passed with 10/10 requirements satisfied. The remaining issues are non-blocking tech
  debt around benchmark-policy noise and workflow hygiene, not requirement failures.
