## Overview

v1.1 shipped one truthful benchmark milestone on top of the v1.0 parity slice: the repo can now
run one canonical Llama-68M generation benchmark in `tools/bench`, execute that benchmark through

benchmark row through the existing snapshot and docs-generation surfaces.

## Milestone Summary

**Decimal Phases:**

- Phase 07.1: Replace the reference-backed decode path with native EMEL decode before any further
  benchmark work (inserted after Phase 7 once the benchmark path was shown to be circular)

**Key Decisions:**

- Reuse the proven v1.0 generation slice in `tools/bench` instead of creating a benchmark-only
  runtime path.
- Insert Phase 07.1 rather than accepting circular benchmark evidence around `llama.cpp` decode.
- Keep the benchmark workflow on the existing `scripts/bench.sh --compare`,
  `--snapshot --update`, `--compare-update`, and `scripts/generate_docs.sh` surfaces.

**Issues Resolved:**

- The canonical benchmark path is no longer circular; both paritychecker and bench now use the
  shared native EMEL backend on the EMEL side.
- Compare mode now enforces the canonical generation row explicitly.
- The canonical generation row is now integrated into the maintained snapshot and generated docs
  surfaces.

**Issues Deferred:**

- The exact low-iteration `scripts/bench.sh --snapshot --compare` smoke check remains noisy at the
  default 10% tolerance.
- The benchmark-policy wrapper still treats benchmark regression drift as non-blocking.
- Tool-local fixture setup still derives some tokenizer/vocab metadata from the explicit
  `llama.cpp` reference model before initialize.

**Technical Debt Incurred:**

- `docs/benchmarking.md` uses an illustrative canonical row example with older sample numbers than
  the refreshed live snapshot/docs outputs.
- `gsd-tools init milestone-op` resolved the current milestone incorrectly during audit, so
  milestone closeout commands should use an explicit version until that helper is fixed.
