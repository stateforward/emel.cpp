# Requirements: EMEL

**Defined:** 2026-04-20
**Core Value:** Prove real end-to-end behavior with explicit SML orchestration and parity-oriented
verification before widening API surface or model scope.

## v1 Requirements

### Reference Backend Architecture

- [x] **GEN-01**: Operator can select a generative reference backend for compare runs without
  changing the EMEL generation lane implementation.
- [x] **GEN-02**: EMEL and every maintained reference backend emit one canonical
  `generation_compare/v1` contract for prompts, generated outputs, verdict metadata, and timing
  fields.
- [x] **ISO-01**: The EMEL generation lane remains isolated from reference-engine model,
  tokenizer, cache, sampler, and runtime objects.

### Reproducible Workload Contract

- [x] **WRK-01**: Operator can run compare workloads from explicit manifests that pin model
  identity, prompt fixture identity, formatter mode, seed, and sampling parameters.
- [x] **WRK-02**: Compare artifacts preserve enough workload metadata to reproduce a run on the
  same engine and explain mismatches across engines.
- [x] **WRK-03**: Workflow rejects or marks non-comparable runs when formatter, tokenization, or
  sampling contracts materially diverge across lanes.

### Maintained Reference Backends

- [x] **REF-01**: At least one maintained non-EMEL generative backend can run through the shared
  compare contract on the canonical generation slice.
- [x] **REF-02**: Backend-specific setup stays confined to the reference lane and does not leak
  into `src/` runtime code or the EMEL compute path.
- [x] **REF-03**: Backend failures surface explicit, reproducible errors without corrupting EMEL
  results or compare summaries.

### Unified Compare Workflow

- [x] **CMP-01**: Operator can run one consistent EMEL-vs-reference generative compare workflow
  regardless of selected backend.
- [x] **CMP-02**: Published artifacts include backend identity, workload manifest identity, output
  summaries, and machine-readable compare verdicts for reproducible review.
- [x] **CMP-03**: Compare publication distinguishes exact-match, bounded-drift, and
  non-comparable outcomes instead of collapsing them into one pass/fail label.

### Proof And Regression

- [x] **PRF-01**: Maintained regression coverage reproduces at least one multi-engine generative
  compare path end to end through the operator-facing workflow.
- [x] **PRF-02**: Stored milestone evidence documents the approved workload boundary and any
  remaining apples-to-oranges caveats for the maintained compare set.

## v2 Requirements

### Follow-On Scope

- **SDK-01**: EMEL exposes a stable public plugin SDK for third-party generation reference
  backends outside the repo.
- **REM-01**: Compare workflows support remote or service-hosted generation engines.
- **MM-01**: The pluggable compare architecture extends to multimodal or non-text generative
  workloads.
- **MAT-01**: One compare workflow can publish a broad maintained model matrix instead of one
  narrow canonical slice.

## Out of Scope

| Feature | Reason |
|---------|--------|
| New public generation C ABI or broad CLI commitments | This milestone is about internal compare architecture, not public API surface |
| Remote HTTP or service-hosted reference engines | Keep the first generative pluggable design local and deterministic |
| Shared model, tokenizer, sampler, cache, or runtime objects between EMEL and reference lanes | Lane isolation remains a hard architectural contract |
| Broad new `src/` runtime support added only to satisfy a reference backend | Reference adapters must not drive EMEL runtime scope creep |
| Embedding or multimodal compare-surface expansion in this milestone | Keep `v1.13` bounded to one maintained generative compare lane |
