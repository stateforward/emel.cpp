# Requirements: EMEL v1.15 ARM Sortformer Diarization GGUF Slice

**Defined:** 2026-04-22
**Core Value:** Prove real end-to-end behavior with explicit SML orchestration and parity-oriented
verification before widening API surface or model scope.

## Goal

Bring up one truthful maintained ARM diarization slice for
`openresearchtools/diar_streaming_sortformer_4spk-v2.1-gguf`, with EMEL-owned loading,
execution, output decoding, parity proof, and benchmark publication. The milestone is intentionally
narrow: one maintained Sortformer GGUF target, one deterministic mono PCM input contract, one
bounded four-speaker diarization output contract, and no broad speech-platform promise.

## v1 Requirements

### Fixture And Model Contract

- [x] **SORT-01**: Maintainer can inspect one pinned fixture contract for the exact
  `openresearchtools/diar_streaming_sortformer_4spk-v2.1-gguf` artifact, including source URL,
  checksum/provenance, expected GGUF architecture metadata, and local test path.
- [x] **SORT-02**: Loader accepts only the maintained Sortformer diarization GGUF contract and
  rejects incompatible, incomplete, or off-scope GGUF files with deterministic errors.
- [x] **SORT-03**: The maintained model contract names the required tensor families and streaming
  parameters needed by EMEL before runtime execution can start.

### Diarization Request And Audio Frontend

- [x] **DIA-01**: Operator can submit deterministic single-channel `float32` PCM at 16,000 Hz
  through a diarization-owned request surface without file decode, resampling, or channel mixing.
- [x] **DIA-02**: Runtime derives the maintained Sortformer acoustic feature/input tensor contract
  natively in EMEL-owned code, with allocation-free dispatch after one-time initialization.
- [x] **DIA-03**: Invalid sample rate, channel count, payload shape, chunk configuration, and output
  buffer capacity fail through explicit diarization errors.

### Sortformer ARM Runtime

- [x] **RUN-01**: EMEL executes the maintained Sortformer encoder/cache/transformer path natively
  on ARM from GGUF tensors without calling llama.cpp, ggml, ONNX Runtime, Python, NeMo, or tool-only
  compute fallbacks from the EMEL lane.
- [x] **RUN-02**: Runtime behavior choices for chunk profile, cache readiness, accepted model
  contract, and error outcomes are modeled in Boost.SML guards/states/transitions, not hidden in
  action/detail branching.
- [x] **RUN-03**: Kernel and tensor work needed for the maintained Sortformer path lands in
  kernel-owned or component-owned execution modules, not ad hoc benchmark or tool helpers.

### Output Contract And Determinism

- [x] **OUT-01**: EMEL emits a deterministic `T x 4` speaker-activity probability matrix where each
  frame corresponds to 0.08 seconds of audio for the maintained profile.
- [x] **OUT-02**: EMEL emits bounded speaker segments with stable `speaker_0` through `speaker_3`
  labels, monotonic timestamps, and explicit overlap behavior derived from the probability matrix.
- [x] **OUT-03**: Repeated runs on the same fixture/audio/profile produce byte-stable JSONL or
  snapshot-friendly diarization records.

### Proof And Benchmark Publication

- [x] **PRF-01**: Repository tests compare EMEL output against a trusted reference baseline for at
  least one canonical multi-speaker audio fixture and preserve the fixture/reference provenance.
- [x] **PRF-02**: Parity proof keeps EMEL and reference lanes separated: EMEL uses only EMEL-owned
  model/runtime state, while any NeMo/llama.cpp/ggml/reference code remains confined to explicit
  reference tooling.
- [x] **BEN-01**: Operator can run one maintained ARM benchmark for the Sortformer diarization slice
  with published timing, fixture identity, profile parameters, and parity/proof status.
- [x] **DOC-01**: Documentation states the exact supported model, input contract, output contract,
  known limitations, and which future Sortformer or diarization work remains out of scope.

## v2 Requirements

Deferred to future milestones.

### Broader Diarization Platform

- **DIA2-01**: Support additional Sortformer GGUF variants beyond the maintained 4-speaker v2.1
  target.
- **DIA2-02**: Add live microphone, streaming socket, or service-hosted diarization APIs.
- **DIA2-03**: Add file decode, resampling, channel mixing, or media metadata ingestion.
- **DIA2-04**: Add ASR transcription, speaker enrollment, speaker embeddings, clustering, or
  multi-model voice-agent orchestration.
- **DIA2-05**: Add CUDA, Metal, Vulkan, CoreML, ONNX, or external accelerator backends.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Generic Sortformer family support | The milestone proves one maintained GGUF slice first. |
| Live service or microphone streaming API | Runtime proof can use deterministic chunked PCM without committing public streaming surface. |
| Media file decode/resample/channel mix | Inference support should not become a media-ingestion milestone. |
| ASR, transcription, or speaker identity enrollment | The target model is diarization only: who spoke when. |
| Tool-only or Python-backed EMEL compute path | Maintained support must be native `src/` execution, not a benchmark scaffold. |
| Whole-tensor dequantize-to-f32 hot-path fallback | Quantized ARM support must remain truthful against the effective operand path. |
| Shared model/runtime/cache objects between EMEL and reference lanes | Lane isolation remains a hard invariant. |
