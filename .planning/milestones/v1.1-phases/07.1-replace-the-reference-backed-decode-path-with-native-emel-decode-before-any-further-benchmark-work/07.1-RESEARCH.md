# Phase 07.1: Replace The Reference-Backed Decode Path With Native EMEL Decode - Research

**Researched:** 2026-03-10
**Domain:** corrective native-decode replacement for the canonical Llama-68M generation slice
**Confidence:** MEDIUM

## Goal

Replace the current reference-backed EMEL decode path with a native EMEL decode path before any
more benchmark compare work, while keeping the milestone narrow to the canonical
`tests/models/Llama-68M-Chat-v1-Q2_K.gguf` slice.

## Replan Trigger

`07.1-01` completed successfully and made the reference seam explicit. Execution then proved the
original `07.1-02` plan was too optimistic: the repo does not yet contain the typed Llama
execution substrate needed to replace the EMEL-side `llama_decode` seam honestly.

Wave 1 remains valid and complete. The replan is only for the remaining work.

## Summary

The repo already has real native orchestration for generation. `src/emel/generator/*` models the
initialize, condition, plan, reserve, compute, sample, render, and flush phases as explicit RTC
actor flows. `src/emel/graph/*` and `src/emel/graph/processor/*` already model reserve/assemble
and validate/prepare/alloc/bind/kernel/extract execution phases.

What is missing is the substrate beneath those phases:

1. typed Llama tensor views derived from loaded `emel::model::data`
2. concrete canonical `model_topology`, `prefill_plan`, and `decode_plan` objects for the
   Llama-68M slice
3. native callback implementations that use those objects instead of forwarding to
   `llama_decode` / `llama_get_logits_ith`

Until those exist, replacing the EMEL-side tool callbacks is not a real native decode fix.

## Current Evidence

### Generator and graph are orchestration shells, not concrete Llama execution

- `src/emel/generator/events.hpp` still accepts `const void * model_topology`,
  `const void * prefill_plan`, and `const void * decode_plan`, plus callback slots for validate,
  prepare, alloc, bind, run, and extract.
- `src/emel/generator/context.hpp` stores those same opaque pointers and callback hooks in
  `graph_binding`.
- `src/emel/generator/actions.hpp` assembles real `emel::graph::event::reserve` and
  `emel::graph::event::compute` requests, but it does not know anything Llama-specific; it passes
  the opaque plans and callback hooks through unchanged.
- `src/emel/graph/events.hpp` and `src/emel/graph/processor/events.hpp` also use `const void *`
  step/topology handles and callback function pointers.

This is the right orchestration seam, but it means native decode requires real objects behind those
handles.

### The assembler and processor are still placeholder-friendly

- `src/emel/graph/assembler/reserve_build_pass/actions.hpp` currently succeeds by copying
  `max_node_count` and `max_tensor_count` into the assembled counts.
- `src/emel/graph/assembler/reuse_decision_pass/actions.hpp` either reuses the reserved topology
  verbatim or copies the request hints into `assembled_node_count` / `assembled_tensor_count`.
- `src/emel/graph/processor/kernel_step/actions.hpp` simply runs the injected callback; the
  processor has no native model-specific compute binding of its own.

So the graph stack is ready to execute a truthful backend, but it is not currently deriving any
canonical Llama topology or step-plan data from `src/`.

### Model loading stops at flat tensors and metadata

- `src/emel/model/data.hpp` stores flat tensor records, names, weights pointers, metadata, and
  hparams, but no typed Llama layer view or canonical decode descriptor.
- `src/emel/model/loader/events.hpp` and `src/emel/model/loader/guards.hpp` only define the parse,
  load, layer-map, and validation callback contract. They do not produce typed execution views.
- The current tool-local layer mapping in `tools/paritychecker/parity_runner.cpp` and
  `tools/bench/generation_bench.cpp` only infers `n_layers` by scanning tensor names; it does not
  build a named tensor-view map for forward compute.

### The current tools still prove the gap directly

- `tools/paritychecker/parity_runner.cpp` still stores dummy integers for `model_topology`,
  `prefill_plan`, and `decode_plan`.
- `tools/bench/generation_bench.cpp` does the same for the benchmark EMEL session.
- Both files still use EMEL-side callbacks that reach `llama_decode` and
  `llama_get_logits_ith`.

## Smallest Missing Native Substrate

The smallest honest substrate that unblocks native EMEL decode is:

1. **Typed canonical Llama weight views**
   - map the loaded flat tensor table into stable named views for token embeddings, final norm,
     output projection, and per-block attention/FFN weights
   - validate required tensor presence, shapes, and canonical metadata against the pinned fixture

2. **Concrete canonical topology and step-plan descriptors**
   - build one real `model_topology` for graph reserve
   - build one real `prefill_plan` and one real `decode_plan` for the canonical request path
   - stop using dummy integers and hint-only assembler counts

3. **Native callback implementation on top of that substrate**
   - validate the request against the typed substrate
   - prepare / allocate / bind / run / extract through native EMEL-owned data instead of the
     reference backend
   - keep the explicit reference path intact for parity and compare mode

Without step 1, there is nothing stable to bind. Without step 2, the graph reserve/compute actors
are still executing placeholder plans. Without step 3, the paritychecker and bench EMEL paths are
still circular.

## Minimum Truthful Replacement Boundary

The minimum honest cut for this corrective phase is:

1. keep `src/emel/generator::sm`, `src/emel/graph::sm`, and
   `src/emel/graph::processor::sm` as the orchestration source of truth
2. add the missing typed substrate beneath their existing event contracts
3. wire paritychecker to that native backend first
4. reuse the same backend in `tools/bench`
5. leave the explicit `llama.cpp` reference path intact and separately observable

This is the smallest change that makes Phase 8 truthful. Anything smaller still benchmarks the
EMEL path through reference decode. Anything larger broadens the milestone beyond the agreed
canonical Llama slice.

## Risks

### 1. Native kernel coverage may still be incomplete

The repo has low-level kernels under `src/emel/kernel/*`, but there is still no proven canonical
Llama forward pass. Substrate work may expose additional missing ops or binding assumptions.

### 2. Graph counts and workspace sizing are currently hint-driven

The assembler reserve/build passes currently trust request hints. Replacing that with real canonical
descriptors may ripple into tests that only exercised placeholder counts.

### 3. State-machine structure changes remain constrained

`AGENTS.md` requires user confirmation before changing state-machine structure. The plan must prefer
typed substrate work under existing event contracts and only escalate if that proves insufficient.

### 4. Bench and parity must share one backend

If paritychecker and bench each grow their own substrate or callback set, Phase 8 will be hard to
trust and harder to maintain.

## Concrete Answer To "What Proves The EMEL Path No Longer Uses `llama_decode` / `llama_get_logits_ith`?"

The proof needs both runtime and static evidence.

### Runtime proof

For each tool, run the EMEL path and the explicit reference path separately with instrumentation:

- EMEL generate path:
  - `reference_decode_calls == 0`
  - `reference_logits_calls == 0`
- Reference generate path:
  - `reference_decode_calls > 0`
  - `reference_logits_calls > 0`

The cleanest way to do this is to route every legitimate reference-only decode call through
tool-local wrappers or counters. Then tests can hard-fail if the EMEL path touches those wrappers.

### Static proof

After the phase, `llama_decode` and `llama_get_logits_ith` should appear only in the explicit
reference implementation, not in the EMEL backend helpers or `src/emel/generator/*`.

### Behavioral proof

The canonical parity flow still has to pass:

- EMEL generation on the canonical fixture succeeds.
- Reference generation on the same fixture succeeds.
- Output/token parity remains correct for the bounded request.

Without all three, the phase is not complete.

### 2. Focused runtime proof after the fix

Recommended focused commands:

- `build/paritychecker_zig/paritychecker --generation --model tests/models/Llama-68M-Chat-v1-Q2_K.gguf --text hello --max-tokens 1`
- `build/bench_tools_ninja/bench_runner --mode=emel | rg "generation/preloaded_request"`

Required assertions:

- EMEL paritychecker generation succeeds without incrementing reference decode/logits counters.
- Reference paritychecker generation still increments those counters.
- EMEL benchmark generation case runs through the native backend only.

### 3. Static seam audit

Use `rg` as a hard check during validation:

- `rg -n "llama_decode|llama_get_logits_ith" tools/bench/generation_bench.cpp tools/paritychecker/parity_runner.cpp src/emel/generator src/emel/graph`

Expected result after the fix:

- Hits remain only in explicit reference-only code paths.
- No hits remain in the EMEL backend callbacks used by generator initialize/generate.

### 5. Exit criteria for Phase 07.1

Phase 07.1 is complete only when all of the following are true:

1. The EMEL generation path in paritychecker no longer calls `llama_decode` or
   `llama_get_logits_ith`.
2. The EMEL generation path in `tools/bench` uses the same native backend as paritychecker.

5. There is automated evidence for the seam, not just a manual code inspection.

## Planning Recommendation

Plan Phase 07.1 as a corrective backend replacement, not as a benchmark feature phase. The safest
approach is:

- keep the existing SML machines intact
- replace the EMEL-side decode backend beneath them
- prove the seam with failing-first tests
- then resume Phase 8 compare work only after the native path is the one being measured

If the implementation effort reveals that real canonical plan/topology objects require structural
changes to `src/emel/generator/*`, that should be treated as an explicit sub-decision during
planning because `AGENTS.md` requires user confirmation before machine-structure changes.
