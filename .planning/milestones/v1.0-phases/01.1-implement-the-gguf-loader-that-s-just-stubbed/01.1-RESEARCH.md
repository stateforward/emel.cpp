<research_summary>
## Summary

`src/emel/gguf/loader/sm.hpp` already defines a clean three-step actor contract:
`probe -> bind_storage -> parse`. The machine, guards, and bound-storage context are real, but the
backend in `src/emel/gguf/loader/detail.hpp` is still a stub:
`probe_requirements(...)` zeroes requirements and `parse_bound_storage(...)` returns success without
reading the file image.

The urgent blocker is therefore narrower than the postponed Phase 2 model-loading work. Phase 1.1
does not need to solve paritychecker integration or whole-model orchestration. It only needs to make
the existing GGUF actor honest:

1. `probe` must parse a real GGUF header and inventory to compute real storage requirements.
2. `bind_storage` must continue to enforce the caller-supplied capacity contract.
3. `parse` must consume the previously bound storage and populate `kv_entry`, KV arena bytes, and
   `emel::model::data::tensor_record` deterministically.

**Primary recommendation:** Keep this phase confined to `src/emel/gguf/loader/*` plus its tests and
fuzz target. Split it into two sequential plans: first implement a real GGUF reader and probe path,
then wire parse to the bound storage context and harden explicit error paths.
</research_summary>

<architecture_patterns>
## Architecture Patterns

### Recommended Project Structure
```text
src/emel/gguf/loader/
├── detail.hpp            # bounded binary reader and GGUF backend helpers
├── actions.hpp           # passes actor-owned context into detail helpers
├── guards.hpp            # capacity and explicit error decisions
└── sm.hpp                # existing probe/bind/parse orchestration

tests/gguf/loader/
└── lifecycle_tests.cpp   # real-file-image and error-path tests

tests/fuzz/
└── gguf_parser_fuzz.cpp  # malformed input coverage
```

### Pattern 1: Detail Helpers Own Binary Parsing
**What:** Perform GGUF binary reading in allocation-free `detail.hpp` helpers that operate over
`std::span<const uint8_t>` and caller-bound storage.
**When to use:** When the actor needs real data-plane work but must keep actions bounded and free of
runtime branching logic.
**Example:** `probe_requirements(...)` walks the GGUF header, KV inventory, and tensor catalog to
produce counts and maximum storage requirements.

### Pattern 2: Actor Context Carries Bound Storage Between Probe and Parse
**What:** Keep bound spans in `action::context` and pass them into parse helpers rather than storing
request-local scratch elsewhere.
**When to use:** When `bind_storage` establishes the caller-owned buffers that `parse` will fill.
**Example:** `actions.hpp` should evolve so `exec_parse` can call a detail helper with
`ctx.kv_arena`, `ctx.kv_entries`, `ctx.tensors`, and `ctx.probed`.

### Pattern 3: Explicit Loader Error Codes
**What:** Map malformed file images, unsupported layout, and insufficient storage into existing
`gguf::loader::error` values.
**When to use:** Whenever parsing or capacity checks fail.
**Example:** Short reads or bad magic -> `parse_failed` or `model_invalid`; bound storage too small
-> `capacity`; null/empty file image -> `invalid_request`.

### Anti-Patterns to Avoid
- **Reference parser leakage:** do not include llama.cpp GGUF implementation in `src/`.
- **Hidden parse state:** do not stash dynamic parse cursors in persistent actor context beyond what
  is needed across `bind_storage` and `parse`.
- **Phase creep:** do not populate `emel::model::data` fields beyond what `gguf::loader` is
  contractually responsible for.
- **Generic success stubs:** do not leave `parse_done` reachable without actual storage population.
</architecture_patterns>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: Probe and parse disagree on capacity
**What goes wrong:** `probe` reports requirements that `parse` later exceeds.
**Why it happens:** Requirement calculation and actual parsing are implemented independently.
**How to avoid:** Share the same bounded reader and sizing rules between probe and parse.
**Warning signs:** `bind_capacity_sufficient` passes, but parse still overruns or fails on valid files.

### Pitfall 2: Parse has no access to bound storage
**What goes wrong:** The actor reaches `parse_done`, but no caller storage is actually filled.
**Why it happens:** `action::exec_parse` currently calls a detail helper with only `file_image`.
**How to avoid:** Plan explicit `actions.hpp` wiring so parse helpers receive the bound spans and
the previously probed requirements.
**Warning signs:** Plan edits only touch `detail.hpp` and never modify the parse action boundary.

### Pitfall 3: Unsupported GGUF values collapse into generic failure
**What goes wrong:** Different malformed inputs all look the same at the error boundary.
**Why it happens:** The parser treats every unexpected value as `internal_error`.
**How to avoid:** Normalize bad magic/version/layout to `model_invalid` or `parse_failed`, and
storage issues to `capacity`.
**Warning signs:** Tests assert only boolean success/failure and never assert specific error classes.

### Pitfall 4: This phase starts populating whole-model semantics
**What goes wrong:** `gguf::loader` starts taking over responsibilities that belong to
`model::loader` or later phases.
**Why it happens:** Parsed tensor metadata naturally invites immediate `model::data` population.
**How to avoid:** Keep Phase 1.1 strictly on the loader's advertised outputs: requirements, KV
storage, tensor descriptors, and explicit parse results.
**Warning signs:** The plan begins modifying `src/emel/model/loader/*`, `tools/paritychecker/*`, or
`src/emel/generator/*`.
</common_pitfalls>

  - malformed-input tests assert explicit `gguf::loader::error` classification
  - fuzz target still builds and runs against non-empty malformed corpora

## Open Questions

1. **How much GGUF metadata must Phase 1.1 parse for the pinned slice?**
   - What we know: probe and parse must be real, not stubbed.
   - What's unclear: whether the first slice needs only tensor/KV inventory or also selected typed
     metadata decoding into richer EMEL-specific representations.
   - Recommendation: keep Phase 1.1 to exact `gguf::loader` outputs; any mapping into higher-level
     `model::data` semantics belongs later.

2. **Should parse fail on unsupported GGUF layouts or preserve partial results?**
   - What we know: the actor already models explicit error outcomes.
   - Recommendation: fail deterministically and leave partial data unclaimed unless the actor
     contract explicitly documents partial success.
