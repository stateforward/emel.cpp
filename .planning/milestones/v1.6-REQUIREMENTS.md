# Requirements: EMEL

**Defined:** 2026-03-27
**Core Value:** Prove real end-to-end behavior with explicit SML orchestration and parity-oriented
verification before widening API surface or model scope.

## v1 Requirements

### Fixture And Conditioning

- [x] **FIX-01**: The repo documents one official `Qwen3-0.6B-Q8_0.gguf` fixture with checksum,
  source, and stable maintained path identity under `tests/models/`.
- [x] **COND-01**: The maintained Qwen3 slice uses one explicit canonical request-conditioning
  contract for both EMEL and `llama.cpp`, derived from the primary GGUF
  `tokenizer.chat_template`, with structured chat-message input and no implicit raw fallback on the
  maintained path.

### Runtime Support

- [x] **RUN-00**: EMEL-owned `src/emel` runtime/kernel code natively executes the canonical
  Qwen3-0.6B slice's blocking `q8_0` projection/output tensors without tool-only or whole-tensor
  dequantize fallback.
- [x] **RUN-01**: EMEL can initialize and generate on the canonical Qwen3-0.6B fixture through the
  maintained generator path in `src/emel`.
- [x] **RUN-02**: The maintained Qwen3 runtime support explicitly handles the canonical slice's
  required `qwen3` topology instead of pretending broad Llama-family compatibility.

### Verification

- [x] **PAR-01**: `tools/paritychecker --generation` proves EMEL against `llama.cpp` on the
  canonical Qwen3 slice using the same fixture and conditioning contract.
- [x] **VER-01**: Regression tests cover the canonical Qwen3 slice and protect the prior
  maintained Llama slice from accidental breakage while support is widened.

### Benchmarking

- [x] **BENCH-01**: `tools/bench` compare output, stored benchmark evidence, and generated docs
  publish one truthful canonical Qwen3 benchmark path aligned with the parity-checked slice.

## v2 Requirements

### Broader Qwen Coverage

- **MODEL-02**: Broaden beyond the canonical Qwen3-0.6B fixture to additional Qwen architectures
  or quantizations once the first slice is proven and benchmarked.

### Prompt Richness

- **COND-02**: Add richer Qwen chat or tool-calling request surfaces only after the canonical
  maintained slice has an explicit and stable conditioning contract.

### Performance

- **GEN-03**: Optimize remaining generator-side hot spots after the canonical Qwen slice is
  correct, parity-backed, and benchmarked.

### Benchmark Policy

- **BENCH-07**: Revisit whether noisy benchmark drift should become a blocking repo gate once the
  maintained compare surfaces are stable enough to justify it.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Broad Qwen-family or multi-fixture support | Keep v1.6 fixed on one official Qwen3-0.6B truth anchor first. |
| Community quant matrix or unofficial Qwen fixtures as milestone truth | The first maintained slice needs one reproducible official provenance chain. |
| Broad new public C API or CLI surface beyond the limited structured-message formatter input approved for the canonical template contract | v1.6 should stay on the existing generator, paritychecker, and benchmark surfaces apart from the approved formatter/request-shape widening needed to represent the primary template honestly. |
| Benchmark gate hardening | The carried benchmark-warning debt is separate from proving the first Qwen slice. |
| Whole-program actor or orchestration rewrites | The milestone is a narrow runtime/model-family bring-up, not a state-machine redesign. |
