# Plan 156-01: Parity Dependency Manifest Gate Closure

<rule_constraints>
- Keep dependency-manifest emission and freshness checks on maintained paritychecker/quality-gate
  entrypoints, not test-only helpers.
- Missing, stale, and uncertain manifest states must force parity gate execution.
- Preserve existing parity mode behavior and CLI validation outside the new manifest operations.
- Keep the manifest deterministic and source-controlled.
</rule_constraints>

## Goal

Make parity dependency manifests operational: operators can emit the maintained manifest, and the
quality gate uses manifest freshness to force full parity coverage when freshness is missing, stale,
or uncertain.

## Tasks

1. Add failing tests for paritychecker CLI manifest emission and manifest freshness decisions.
2. Add quality-gate source checks for production manifest emission, baseline comparison, and
   uncertain-state escalation.
3. Implement paritychecker CLI manifest operations backed by `dependency_manifest::*`.
4. Add a maintained manifest baseline and update manifest docs with CLI and gate examples.
5. Wire `scripts/quality_gates.sh` so stale, missing, or uncertain manifest data forces parity.
6. Run focused paritychecker tests, direct manifest CLI smoke checks, and changed-file scoped
   quality gates.
