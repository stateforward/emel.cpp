# Plan 149-01: Parity Engine Adapter Split

<objective>
Move existing parity mode implementations behind explicit runner-facing engine adapter entrypoints
so `parity_runner.cpp` becomes orchestration-only while existing mode behavior stays stable.
</objective>

<rule_constraints>
## Rule Traceability

- `PARITY-02` requires the runner to stop owning bulk per-mode implementation. Tasks 2 and 3 move
  the implementation bundle behind adapter entrypoints and recreate the runner as orchestration.
- `ENGINE-01` requires explicit engine interfaces/adapters. Tasks 1 and 2 introduce the adapter
  contract and named mode entrypoints.
- Lane isolation is preserved by moving source ownership only; existing EMEL/reference model and
  runtime objects remain in the mode implementation bundle and are not shared.
</rule_constraints>

<tasks>
<task id="1" type="auto">
<title>Add runner-facing engine adapter contract</title>
<read_first>
- `tools/paritychecker/parity_runner.hpp`
- `tools/paritychecker/parity_runner.cpp`
</read_first>
<instructions>
Add a small parity engine adapter header/source that names each engine by `parity_mode`, name, and
run function. Provide a lookup helper for `run_parity(...)`.
</instructions>
<acceptance_criteria>
- Tests can inspect adapter names for tokenizer, GBNF, kernel, Jinja, and generation.
- Adapter lookup returns no engine for invalid mode values.
</acceptance_criteria>
</task>

<task id="2" type="auto">
<title>Move existing mode implementation behind adapter entrypoints</title>
<read_first>
- `tools/paritychecker/parity_runner.cpp`
- `tools/paritychecker/tokenizer_parity.hpp`
</read_first>
<instructions>
Move the current bulk implementation out of `parity_runner.cpp` into an engine implementation file.
Expose one adapter entrypoint per existing mode that calls the existing implementation without
changing behavior.
</instructions>
<acceptance_criteria>
- Tokenizer, GBNF, kernel, Jinja, and generation modes have named adapter entrypoints.
- Existing per-mode implementation details remain behaviorally unchanged.
</acceptance_criteria>
</task>

<task id="3" type="auto">
<title>Reduce runner to orchestration</title>
<read_first>
- `tools/paritychecker/parity_runner.cpp`
- `tools/paritychecker/CMakeLists.txt`
</read_first>
<instructions>
Recreate `parity_runner.cpp` as a small orchestration file that resolves the engine adapter for
`opts.mode` and invokes it. Update CMake so the new adapter and implementation files build into
the paritychecker executable and tests.
</instructions>
<acceptance_criteria>
- `parity_runner.cpp` no longer includes llama/ggml/jinja/generator/kernel implementation headers.
- `parity_runner.cpp` contains no bulk mode implementation functions.
</acceptance_criteria>
</task>
