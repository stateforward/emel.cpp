# Plan 151-01: Parity Dependency Manifest Emission

<objective>
Add a deterministic source-backed parity dependency manifest surface that covers runner source,
config, fixture, model, script, and snapshot inputs for later quality-gate impact selection.
</objective>

<tasks>
<task id="1" type="auto">
<title>Add manifest API</title>
<instructions>
Create a paritychecker manifest module with typed dependency records, per-mode record lookup,
deterministic rendering, file writing, and a freshness predicate for conservative gate selection.
</instructions>
<acceptance_criteria>
- Manifest records are grouped by registered parity mode.
- Missing, stale, or uncertain freshness returns full-gate required.
- Rendering is deterministic.
</acceptance_criteria>
</task>

<task id="2" type="auto">
<title>Cover parity dependency inputs</title>
<instructions>
Populate records for tokenizer, GBNF, kernel, Jinja, and generation modes covering source, config,
fixture, model, script, and snapshot inputs where applicable.
</instructions>
<acceptance_criteria>
- Every registered engine has non-empty manifest records.
- Records include `tools/paritychecker/reference_ref.txt` and `scripts/quality_gates.sh`.
- Maintained generation fixture/model/snapshot inputs are represented.
</acceptance_criteria>
</task>

<task id="3" type="auto">
<title>Document and test manifest format</title>
<instructions>
Document `parity_dependency_manifest/v1`, add focused doctests for record coverage, conservative
freshness semantics, deterministic rendering, file writing, and CMake registration.
</instructions>
<acceptance_criteria>
- Tests fail if a registered mode has no manifest records.
- Tests fail if freshness uncertainty becomes permissive.
- Tests fail if the manifest source is dropped from the shared paritychecker target.
</acceptance_criteria>
</task>

<task id="4" type="auto">
<title>Validate and record Phase 151</title>
<instructions>
Run focused paritychecker builds/tests and the changed-file scoped quality gate. Restore any
unapproved timing snapshot churn.
</instructions>
<acceptance_criteria>
- `paritychecker` and `paritychecker_tests` build.
- `paritychecker_tests` passes.
- Changed-file scoped quality gate passes.
</acceptance_criteria>
</task>
</tasks>

</verification>

- Per-mode manifest records exist for all registered parity engines.
- The manifest covers source, config, fixture, model, script, and snapshot inputs.
- Missing, stale, or uncertain data is a full-gate trigger.
- Manifest format and generation are documented and tested.
</success_criteria>
