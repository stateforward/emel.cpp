# Phase 12.1: Enforce SML Tensor Lifecycle Orchestration - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning
**Source:** User-inserted urgent phase directive

<domain>
## Phase Boundary

Phase 12.1 corrects an architectural violation in the shipped EMEL runtime: the existing
`emel::tensor::sm` / `sml::utility::sm_pool` tensor lifecycle machinery exists under `src/emel/tensor`
but is not wired into the graph execution pipeline. This phase must integrate that existing tensor
state machine into graph reservation, kernel execution publication, and liveness-based release so
tensor lifecycle truth is formally tracked through Boost.SML rather than inferred from raw pointers,
`tensor_view`, or `tensor_record` math.

This phase is architecture repair inside the shipped runtime. It is not a new user-facing feature,
not a benchmark phase, and not a request to widen the public API. The result should be that graph
and generator execution depend on tensor lifecycle truth surfaced by `emel::tensor::sm` while
preserving the zero-allocation hot-path contract and the existing data-oriented pool performance
profile.

</domain>

<decisions>
## Implementation Decisions

### Scope And Ownership
- **D-01:** The existing `emel::tensor::sm` and its underlying `tensor_pool` are the required
  lifecycle authority for this phase; the phase must not introduce a second parallel tensor state
  tracker.
- **D-02:** Tensor lifecycle orchestration must be wired through the graph-owned runtime path
  (`graph`, `graph::assembler`, `graph::allocator`, `graph::processor`) instead of remaining an
  optional or tool-local side surface.
- **D-02a:** This phase is a hard cutover. The current ad hoc tensor lifecycle behavior in graph
  and generator must be removed or rendered unreachable once the SML-owned lifecycle path lands;
  do not leave a dual-path fallback that preserves the existing bypass implementation.
- **D-03:** The phase may change Boost.SML machine structure as needed because the user explicitly
  requested this architectural correction.
- **D-04:** Generator work in this phase is only to the extent required to route the existing core
  inference pipeline through the corrected graph/tensor orchestration seam; this is not a request
  to redesign generation semantics beyond that.

### Tensor Lifecycle Contract
- **D-05:** Graph reservation/build work must formally dispatch tensor reservation into
  `emel::tensor::sm` for every active graph tensor once topology and memory assignment are known.
- **D-06:** After a kernel successfully produces tensor contents, the processor must formally
  dispatch `publish_filled_tensor` for the produced tensor(s) rather than relying on implicit
  callback success as proof of readiness.
- **D-07:** When tensor liveness is exhausted and memory becomes reusable, the processor must
  formally dispatch `release_tensor_ref` before reuse is treated as valid.
- **D-08:** The final architecture must mathematically guarantee that no kernel step executes until
  its required input tensors are known-filled according to the tensor pool.
- **D-09:** The final architecture must mathematically guarantee that tensor memory is not reused
  until the tensor pool has recorded release for the prior lifecycle epoch.
- **D-10:** If the current graph processor callbacks do not expose enough tensor identity data to
  publish and release lifecycle transitions truthfully, this phase must add the minimum necessary
  runtime information to the internal graph pipeline to make those transitions explicit.

### Performance And Safety Constraints
- **D-11:** The zero-allocation constraint on the hot inference path remains mandatory.
- **D-12:** The implementation must use the existing `sm_pool` mechanics so tensor lifecycle
  tracking stays cache-contiguous and data-oriented; do not replace it with per-tensor heap-backed
  machines or dynamic containers in dispatch paths.
- **D-13:** The phase must preserve RTC/no-queue SML semantics from `docs/rules/sml.rules.md`;
  tensor lifecycle dispatch must remain synchronous and must not introduce mailbox/deferred-event
  patterns.
- **D-14:** Runtime control flow must stay explicit in SML guards/states/transitions, not hidden in
  callback-side branching or ad-hoc loop-based routing logic.

### Verification Contract
- **D-15:** “Done” for this phase requires tests that prove both lifecycle state progression and the
  new execution gating, not just successful builds.
- **D-16:** Tests must demonstrate that kernels are blocked or fail deterministically when required
  tensors are not yet published as filled.
- **D-17:** Tests must demonstrate that tensor reuse is blocked until release is formally recorded
  through the tensor pool.

### the agent's Discretion
- Exact internal event shapes or callback payload changes needed to carry tensor IDs and lifecycle
  data across graph phases.
- Exact placement of reservation dispatch between assembler and allocator subphases, as long as
  reservation happens only after topology/memory assignment is known and remains graph-owned.
- Exact test decomposition across `tests/graph/*`, `tests/tensor/*`, and `tests/generator/*`, as
  long as lifecycle truth and execution gating are both covered.

</decisions>

<specifics>
## Specific Ideas

- The current tensor lifecycle API is already present and narrow:
  `reserve_tensor`, `publish_filled_tensor`, `release_tensor_ref`, `reset_tensor_epoch`, and
  `capture_tensor_state`.
- The current gap is architectural, not missing primitives: `tensor::action::context` already owns
  `detail::tensor_pool`, and `tensor::actions.hpp` already dispatches indexed pool operations via
  `process_indexed`.
- The likely runtime shape is that graph-owned context will need access to a tensor actor or tensor
  pool handle, and graph callbacks will need enough tensor identity information to publish/release
  lifecycle state truthfully after bind/kernel/extract/liveness phases.
- Current processor steps are callback-driven (`validate`, `prepare_graph`, `alloc_graph`,
  `bind_inputs`, `run_kernel`, `extract_outputs`) with no lifecycle coupling beyond generic success
  or failure flags. This phase should make tensor lifecycle an explicit orchestration dependency.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `tensor::sm` already provides exactly the lifecycle verbs this phase needs. The work is
  integration and orchestration, not new tensor-pool invention.
- `tensor::view` already demonstrates a pattern for reading lifecycle state through a dedicated SML
  surface rather than peeking directly into pool storage.
- The graph processor already has explicit phased orchestration (`validate -> prepare -> alloc ->
  bind -> kernel -> extract`) that can host lifecycle gates without abandoning Boost.SML.
- The user explicitly rejected a transitional bridge design: planning must target removal of the
  ad hoc lifecycle bypass rather than an additive “new path plus legacy fallback” implementation.

### Established Patterns
- Graph, assembler, allocator, and processor all already use destination-first Boost.SML tables and
  callback-fed internal phase outcome tracking.
- Current graph/processor design passes generic callbacks and opaque `compute_ctx` pointers through
  the pipeline, which suggests this phase may need new internal typed runtime data rather than more
  pointer-driven implicit state.
- Current graph state machines do not own any tensor actor in context, so tensor lifecycle truth is
  necessarily bypassed today.

### Integration Risks
- The current graph processor event contract only models generic phase callbacks, not per-tensor
  lifecycle facts. Planning must account for how tensor IDs and produce/consume information move
  through the pipeline without violating the no-dispatch-local-context rules.
- Reservation placement must stay truthful: it cannot happen before topology/allocation information
  is known, but it also cannot be delayed so far that kernel/input gating becomes unenforceable.
- The current roadmap execution order now logically becomes `12 -> 12.1 -> 13`, even though the
  roadmap prose inserted the urgent phase after initial milestone completion.

</code_context>
