# Phase 12.1: Enforce SML Tensor Lifecycle Orchestration - Research

**Researched:** 2026-03-21
**Status:** Ready for planning

## Research Question

How should EMEL cut over from its current ad hoc graph/generator tensor lifecycle behavior to the
existing `emel::tensor::sm` / `sml::utility::sm_pool` machinery without leaving the bypass path in
place, while preserving RTC semantics and zero-allocation hot-path behavior?

## Current Architecture

### What Already Exists

- `src/emel/tensor/detail.hpp` already defines `tensor_pool` as
  `boost::sml::utility::sm_pool<tensor_storage, tensor_router_model>`.
- `src/emel/tensor/actions.hpp` already exposes the required lifecycle verbs through indexed pool
  operations:
  - `reserve_tensor` -> `op_reserve`
  - `publish_filled_tensor` -> `op_publish_filled`
  - `release_tensor_ref` -> `op_release_ref`
  - `reset_tensor_epoch` -> `op_reset_epoch`
  - `capture_tensor_state` -> direct state readback
- `src/emel/tensor/sm.hpp` already wraps those operations in a normal SML actor surface.
- `tests/tensor/lifecycle_tests.cpp` proves the current lifecycle semantics:
  - compute tensors go `unallocated -> empty -> filled -> empty`
  - leaf tensors go `unallocated -> leaf_filled` and release/reset are no-ops
  - invalid lifecycle transitions become `internal_error`
- `src/emel/tensor/view/*` and `tests/tensor/view/lifecycle_tests.cpp` already provide a
  state-query pattern that can be reused when graph code needs filled-state truth.

### What Is Currently Bypassing It

- `src/emel/graph/context.hpp` owns only `assembler::sm` and `processor::sm`; there is no tensor
  actor in graph-owned context.
- `src/emel/graph/processor/events.hpp` models only generic callback phases:
  `validate`, `prepare_graph`, `alloc_graph`, `bind_inputs`, `run_kernel`, `extract_outputs`.
  No tensor lifecycle data is carried through the execution pipeline.
- `src/emel/graph/processor/sm.hpp` moves from `bind_step` directly to `kernel_step` based only on
  generic bind success, not on tensor filled-state truth.
- `src/emel/graph/assembler/*` and `src/emel/graph/allocator/*` currently produce only topology and
  count/capacity outputs, not per-tensor lifecycle reservation descriptors.
- `src/emel/generator/detail.hpp` implements the actual runtime callbacks for
  `prepare_graph`, `alloc_graph`, `bind_inputs`, `run_kernel`, and `extract_outputs`. Those
  callbacks manipulate backend buffers and tensor-like views directly, but do not reserve/publish/
  release lifecycle state through `emel::tensor::sm`.

## Constraints That Matter

- Hard cutover only. Do not leave the old ad hoc lifecycle path standing as a fallback.
- No mailbox, defer queue, or other non-RTC machinery.
- No allocation on the hot inference path.
- Runtime control flow must remain explicit in SML states/guards/transitions.
- Graph/generator orchestration remains the source of truth; do not push lifecycle decisions into
  opaque callback-side branching.

## Important Technical Finding

### Reservation Cannot Be Truthfully Assembler-Only In The Current Design

The user directive asks for reservation when graph topology is built and memory offsets are
assigned. Current assembler/allocator code does not yet carry enough runtime information to reserve
truthfully:

- assembler outputs only `graph_topology`, counts, required bytes, version, and reuse flags
- allocator outputs only counts and required bytes
- neither phase currently knows the actual bound runtime buffer pointers used by the generator
  backend

That means a truthful hard cutover likely needs two pieces:

1. **Graph planning metadata**
   - assembler/allocator must define which tensors are active and what their lifecycle schedule is
   - this gives the graph processor enough identity and liveness truth to orchestrate reserve/
     publish/release

2. **Runtime reservation dispatch**
   - actual `reserve_tensor` dispatch should happen only once the execution pipeline has the real
     runtime buffer for each tensor, which today appears no earlier than the bind/allocation side of
     processor execution

This is still consistent with the directive if assembler/allocator become the source of reservation
metadata while processor performs the actual reserve dispatch at the earliest truthful point.

## Recommended Hard-Cutover Architecture

### 1. Graph Owns Tensor Lifecycle Authority

- Add `emel::tensor::sm` ownership to graph-owned context, not just generator or tool-local code.
- Inject that tensor actor into the submachines that need lifecycle orchestration:
  graph, graph::assembler, graph::processor, and any supporting tensor-state reader if needed.
- Do not let generator directly own an alternate lifecycle tracker.

### 2. Make Tensor Lifecycle Metadata Explicit In The Graph Runtime

The current processor callback interface is too generic. A hard cutover needs internal graph-owned
runtime metadata that identifies:

- which tensor IDs are inputs for a node/phase
- which tensor IDs are outputs
- which tensors are leaf/bound versus compute-produced
- consumer ref counts or release schedule
- where the actual buffer and size for a tensor becomes available

This metadata likely belongs in graph internal events / graph execution runtime data, not in
persistent machine context.

### 3. Add Explicit Lifecycle Phases To Processor Orchestration

The current processor flow:

`validate -> prepare -> alloc -> bind -> kernel -> extract`

is insufficient for lifecycle truth. A plausible hard-cutover flow is:

`validate -> prepare -> alloc -> bind/reserve -> input-ready-gate -> kernel -> publish-outputs -> release-expired -> extract`

The exact naming can vary, but the cutover needs explicit states for:

- reserve active tensors once real bindings exist
- verify required inputs are filled before kernel dispatch
- publish produced tensors after successful kernel completion
- release tensors whose liveness has ended before reuse is considered valid

### 4. Use Existing Tensor State Semantics, Do Not Invent New Ones

Current pool semantics are already suitable:

- compute tensors start `empty`
- they become `filled` only via publish
- releases decrement refs until final release returns them to `empty`
- invalid transitions surface `internal_error`

Planning should reuse those semantics directly instead of adding a second status vocabulary in graph
context.

## Guarding Kernel Execution

To satisfy the user's definition of done, kernel execution must become impossible unless required
inputs are known-filled by the tensor authority.

There are two viable mechanisms:

1. **State-query gate**
   - processor explicitly captures tensor state before kernel dispatch using `capture_tensor_state`
     or `tensor::view`
   - guards branch on “all required inputs filled”

2. **Precomputed readiness event**
   - processor performs state capture in an explicit action/state phase and publishes a typed
     internal event with the result
   - kernel transition is guarded on that typed result

Given the repo rules against runtime branching in actions and the existing SML phase pattern, the
second option is safer: capture in a dedicated phase, branch via explicit guards/states.

## Reuse / Release Implications

The tensor pool already models release-to-empty semantics, but current graph reuse logic is based on
topology reuse and generic success flags. A hard cutover therefore needs reuse decisions to depend
on formal release, not just prior graph completion.

That likely means:

- lifecycle release must happen before any reused tensor slot is considered available
- reset/epoch handling may be required at graph-start or sequence-start boundaries for compute
  tensors that persist across dispatches
- leaf tensors must remain distinguishable from compute tensors so the pool's leaf no-op semantics
  stay correct

## Recommended Plan Decomposition

### Plan 01: Introduce Graph-Owned Tensor Lifecycle Runtime

Focus:
- inject tensor actor ownership into graph orchestration
- add internal graph/runtime metadata for tensor lifecycle identity and scheduling
- wire truthful reserve dispatch at the earliest point where runtime buffers are known

Why first:
- everything else depends on tensor lifecycle data being available in graph execution

### Plan 02: Cut Processor Over To Lifecycle-Gated Execution

Focus:
- add explicit processor phases/states for readiness gate, publish, and release
- make kernel dispatch depend on filled-state truth
- make reuse depend on formal release

Why second:
- this is the core mathematical guarantee the user asked for

### Plan 03: Remove The Bypass From Runtime Callers And Regress

Focus:
- update generator/runtime callers to the new graph/tensor contract
- remove or render unreachable the ad hoc lifecycle path
- expand tests and run repo gates

Why third:
- caller cutover should happen only after the graph/tensor orchestration seam is real

## Risks

- Existing graph test doubles are heavily generic and may need richer fake lifecycle payloads.
- The current graph callback API may be too narrow; plan scope must allow internal event/request
  reshaping to carry tensor IDs and buffer bindings truthfully.
- If planning keeps lifecycle state in machine context rather than typed runtime events, it will
  violate the repo's context rules.
- If planning tries to reserve tensors before runtime buffers exist, it will produce a formally
  incorrect “SML managed” path that is still lying.

## Validation Architecture

### Required automated proof

- tensor lifecycle tests proving reservation/publish/release/reset semantics still hold
- graph/processor tests proving kernel dispatch is blocked when input tensors are not filled
- graph/processor tests proving reuse/release ordering is enforced
- generator/runtime tests proving the hard cutover path is the only active path
- repo regression via `ctest --test-dir build/debug --output-on-failure -R emel_tests` and
  `scripts/quality_gates.sh`

### Suggested fast feedback loop

- targeted `emel_tests` runs after each task group
- full `scripts/quality_gates.sh` after each plan wave

## Planning Recommendation

Proceed with a 3-plan phase:

1. graph-owned tensor actor injection plus lifecycle reservation metadata
2. processor lifecycle gating/publish/release cutover
3. generator/runtime caller cutover plus regression proof

This is the smallest plan set that still satisfies the user's hard-cutover requirement and the
definition of done.
