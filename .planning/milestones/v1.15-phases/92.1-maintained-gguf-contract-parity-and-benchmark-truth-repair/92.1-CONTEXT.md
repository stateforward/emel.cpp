# Phase 92.1: Maintained GGUF contract parity and benchmark truth repair - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Repair the parity and benchmark publication lane so v1.15 proves the real maintained Sortformer
engine path end to end after the runtime contract repair lands: pinned GGUF artifact, real
loader-built execution contract, maintained pipeline actor, one maintained real-audio fixture, and
a truthful ARM baseline/report.

</domain>

<decisions>
## Implementation Decisions

### Maintained Provenance
- The EMEL parity and benchmark lanes must load the maintained model from
  `tests/models/diar_streaming_sortformer_4spk-v2.1.gguf`.
- The execution contract must be built by the real `src/emel/model/sortformer` path, not by an
  in-tool fabricated `make_contract(...)` helper.
- Any remaining synthetic contract/fixture path feeding parity or benchmark is a blocker, not an
  acceptable interim publication lane.

### Maintained Audio Fixture
- The phase should use one pinned real-audio fixture rather than synthetic generated PCM.
- Prefer a public upstream/model-card-aligned sample when a concrete downloadable example exists.
- If upstream does not publish a canonical sample, pick one public multi-speaker audio fixture,
  document provenance explicitly, and make it the maintained benchmark/parity clip.

### Benchmark Truth
- The primary optimization and publication target is the steady-state maintained inference lane, not
  one-time setup work.
- One-time costs such as model loading, fixture decode/bind/setup, or first-time initialization may
  be reported separately, but they are secondary metrics.
- The phase should establish one honest ARM baseline on exact maintained model + maintained audio
  fixture + maintained runtime path before claiming any further optimization wins.

### Reference Lane Isolation
- Reference parity may remain a separate lane, but it must not share EMEL-owned model/runtime/cache
  objects.
- Parity, benchmark, and docs must only claim what the maintained EMEL lane actually proves.

### Dependency Boundary
- Phase `92.2` owns the deeper `src/` runtime contract repair.
- Phase `92.1` owns maintained parity, benchmark publication, docs, and audit evidence on top of
  that repaired runtime.

### the agent's Discretion
- The agent may choose the exact maintained public audio sample and exact steady-state benchmark
  slicing/reporting format as long as provenance, reproducibility, and maintained-path truth are
  explicit.

</decisions>

<specifics>
## Specific Ideas

- Publish steady-state inference timing as the primary ARM metric; report model load and other
  one-time setup costs separately.
- Use the real maintained GGUF model and a real maintained audio sample for both parity and
  benchmark publication.
- Keep any baseline claims honest: if no external ARM baseline exists, establish the repo's first
  truthful baseline on exact hardware rather than inventing a target.
- Consume the runtime produced by Phase `92.2`; do not reintroduce tool-local compatibility
  shims, synthetic contracts, or synthetic PCM to get publication green.

</specifics>
