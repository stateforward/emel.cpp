# Phase 92.6 Plan 1: Resolve Sortformer PyTorch Parity And ONNX Benchmark Drift

## Goal

Root-cause the real Sortformer output drift using PyTorch/NeMo as the parity reference lane and
ONNX Runtime as the benchmark reference lane. Either land the kernel/runtime fix needed for exact
PyTorch parity and ONNX benchmark-reference agreement, or produce a source-backed contract decision
explaining any ONNX limitation separately from the parity target.

## Tasks

1. Wire and run the PyTorch/NeMo parity reference lane.
   - Use the official model-card `SortformerEncLabelModel` inference path on the maintained WAV
     fixture.
   - Label the backend as `pytorch.nemo.sortformer.v2_1` with reference role `parity_reference`.
   - Recreate the Python environment reproducibly with `uv` and a pinned requirements file.
   - Time `diarize(...)` execution only; exclude one-time model load.

2. Inspect ONNX graph and model-card execution contract.
   - Verify feature scale/layout assumptions, input names, cache/fifo length semantics, output
     logits window, and speaker order.
   - Label the ONNX backend as `onnx.sortformer.v2_1` with reference role `benchmark_reference`.
   - Cross-check ONNX output against PyTorch before using ONNX as a closeout target.

3. Add targeted probe artifacts, not ad hoc parity fallbacks.
   - Emit deterministic intermediate facts from the ONNX lane and EMEL lane only for diagnosis.
   - Keep probes under `tools/bench` or `build/` and avoid widening production APIs unless the
     runtime needs a real maintained inspection surface.

4. Isolate drift to one contract layer.
   - Compare PyTorch, ONNX, and EMEL probabilities/segments, then feature tensors, logits,
     segment decoding, cache behavior, and model artifact identity in that order.
   - If drift is caused by native EMEL arithmetic, fix the owning kernel/runtime code in `src/`.
   - If drift is caused by an ONNX artifact/input-contract mismatch, update compare metadata and
     milestone docs so the limitation is explicit and source-backed.

5. Verify final parity or final scoped limitation.
   - Rerun the self-recorded regression snapshot compare.
   - Rerun the PyTorch parity compare on the real model path.
   - Rerun the ONNX benchmark compare on the real local model.
   - Update docs, benchmark reporting, phase summary, verification, `STATE.md`, `ROADMAP.md`, and
     milestone audit to reflect the result.

6. Treat PyTorch parity drift as a hard failure.
   - `diarization_compare.py` must return nonzero when `reference_role=parity_reference` is not an
     exact match.
   - ONNX remains the `benchmark_reference` lane and can report bounded drift separately.
