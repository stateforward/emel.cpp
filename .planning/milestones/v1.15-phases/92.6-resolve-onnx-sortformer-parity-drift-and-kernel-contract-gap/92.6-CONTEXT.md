# Phase 92.6 Context: Resolve Sortformer PyTorch Parity And ONNX Benchmark Drift

## Trigger

Phase `92.5` added a real ONNX Runtime reference lane for the maintained Sortformer diarization
fixture. The lane ran successfully, but it did not exact-match EMEL:

- EMEL/self-recorded regression snapshot: `output_dim=71`, checksum `2496121829252312372`
- ONNX Runtime lane: `output_dim=17`, checksum `4249677247906920305`
- PyTorch/NeMo parity lane: `output_dim=17`, checksum `4249677247906920305`
- Summary status: `bounded_drift reason=output_mismatch`

This means the benchmark surface is now truthful, but the milestone cannot be closed as parity. The
recorded baseline was produced by EMEL, so it is useful only as a regression snapshot, not as an
independent correctness oracle. PyTorch/NeMo must be the parity reference lane because it is the
official model-card execution path. ONNX must be treated as the benchmark reference lane and
cross-checked against PyTorch before ONNX output is used as a closeout target.

## Candidate Drift Causes

- The ONNX model card confirms the export is configured for `chunk_len=1504`, `fifo_len=0`,
  `spkcache_len=188`, `feat_dim=128`, and `emb_dim=512`. Feature and pre-encoder parity make the
  frontend unlikely to be the primary drift source.
- The ONNX output contract is probability output, not raw logits. This was fixed in the reference
  runner, but exact parity remains open.
- The remaining drift localizes to transformer execution after pre-encoder parity and before the
  speaker probability head.
- The ONNX artifact may not be weight-equivalent to the checked-in GGUF artifact.
- EMEL native arithmetic may still differ from the exported model path in a kernel-owned operation.
- PyTorch/NeMo may confirm whether ONNX is weight/contract equivalent to the official model-card
  path. If PyTorch and ONNX disagree, ONNX remains only a benchmark reference with an explicit
  limitation until the ONNX contract is reconciled.

## Constraint

Do not solve this by hiding the PyTorch or ONNX lanes, replacing them with the recorded baseline, or
adding tool-local compute that bypasses the maintained runtime. Any EMEL-side defect must be fixed
in the owning `src/` kernel/runtime path.

## Lane Roles

- `pytorch.nemo.sortformer.v2_1`: parity reference lane using
  `SortformerEncLabelModel.diarize(..., include_tensor_outputs=True)` on the maintained WAV
  fixture. This is the correctness oracle for closeout. Its runtime environment is recreated with
  `uv` from `tools/bench/diarization_pytorch_reference_requirements.txt`.
- `onnx.sortformer.v2_1`: benchmark reference lane using ONNX Runtime CPU and the maintained
  feature tensor. This is the performance/bench comparator and must match PyTorch before its output
  is treated as a correctness target.
- `recorded.diarization.baseline`: self-recorded regression snapshot only.
