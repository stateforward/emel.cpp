**Gathered:** 2026-04-22

<domain>
## Phase Boundary

Phase 83.1 owns the first native execution slice after the Phase 83 contract repair: the maintained
Conformer encoder contract under `src/emel/diarization/sortformer/encoder/`. It should bind the
real `enc.*` tensor families for the one maintained GGUF profile and land fixed-profile native
encoder kernel surfaces without creating a generic runtime dispatcher.

</domain>

<decisions>
## Implementation Decisions

### Stage Ownership
- Place encoder-owned code under `src/emel/diarization/sortformer/encoder/`.
- Keep feature extraction in `encoder/feature_extractor/`; do not rename it back to ambiguous
  `frontend` or move it into a generic runtime.
- Do not create placeholder modules/cache/transformer source files in this phase.

### Maintained Tensor Contract
- Treat `enc.pre.*` plus `enc.l0.*` through `enc.l16.*` as the maintained encoder contract.
- Validate tensor names and fixed maintained dimensions instead of accepting any arbitrary encoder
  family.
- Keep tensor binding as one-time setup/readiness work, not as a per-dispatch runtime search.

### Performance Direction
- Use fixed-profile scalar kernels first, then let ARM benchmarks identify AArch64 kernels worth
  moving into `src/emel/kernel/aarch64`.
- Keep data-plane loops inside component-owned detail kernels and keep behavior selection in guards
  and transition rows when an actor is introduced.

### the agent's Discretion
- The exact first encoder kernel surface is at the agent's discretion, but it must be truthful:
  component-owned numeric work over validated maintained shapes, not a synthetic substitute for the
  full encoder stack.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `emel::model::tensor_name_view` exposes GGUF tensor names without copying.
- `src/emel/model/sortformer/detail.*` already validates top-level Sortformer tensor families.
- `src/emel/diarization/sortformer/encoder/feature_extractor/detail.hpp` owns PCM-to-feature
  preparation constants for the maintained profile.

### Established Patterns
- Component files use canonical bases such as `detail.hpp` and `detail.cpp`.
- One-time contract binding can allocate or scan model tensors; dispatch actions must stay bounded
  and allocation-free.
- Tests are scoped by domain and machine/component.

### Integration Points
- Phase 83.1 feeds later `modules/cache` and `transformer` phases by providing a validated encoder
  contract and deterministic native kernel primitives.

</code_context>

  position tensors with maintained f32 shapes.
