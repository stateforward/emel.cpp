# Plan 83.5-01: Native Feature-To-Encoder Frame Path

<objective>
Derive the executor's `188 x 512` encoder-frame input from the maintained `188 x 128` feature
tensor through EMEL-owned Sortformer encoder code, then prove the existing executor consumes the
native output unchanged.
</objective>

<rule_constraints>
- Use maintained `enc.pre.*` tensor contracts already bound by `encoder::detail::bind_contract`.
- Keep numeric implementation in `src/emel/diarization/sortformer/encoder/`.
- Do not allocate during feature-to-encoder compute.
- Do not change modules/cache/transformer execution semantics.
- Add focused doctest coverage under `tests/diarization/sortformer/encoder/lifecycle_tests.cpp`.
</rule_constraints>

<tasks>
1. Extend `encoder::detail` with fixed-profile constants, caller-owned workspace, and a
   `compute_encoder_frames_from_features` API.
2. Implement the native pre-encoder slice using maintained `enc.pre.*` tensor data and fixed
   `188 x 128 -> 188 x 512` shapes.
3. Add lifecycle tests for deterministic encoder-frame derivation, invalid feature/output shape
   rejection, and executor composition from native encoder frames.
4. Run focused diarization tests and the required quality gate.
</tasks>
