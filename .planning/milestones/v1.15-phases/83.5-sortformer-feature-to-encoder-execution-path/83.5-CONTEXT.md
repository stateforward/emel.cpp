**Gathered:** 2026-04-23

**Mode:** Autonomous smart discuss defaulted by prior user direction

<domain>
## Phase Boundary

Phase 83.5 owns the native bridge between the prepared Sortformer acoustic feature tensor and the
executor's validated `188 x 512` encoder-frame input. The phase must remove the remaining synthetic
feature-to-encoder projection gap without changing the Phase 83.4 modules/cache/transformer
execution behavior.

</domain>

<decisions>
## Implementation Decisions

### Stage Ownership
- Put feature-to-encoder numeric work under `src/emel/diarization/sortformer/encoder/`.
- Keep the executor's modules/cache/transformer path behaviorally unchanged.
- Do not add a generic Sortformer runtime, dispatcher, or tool-only compute lane.

### Maintained Tensor Contract
- Consume the maintained `enc.pre.*` tensor family that Phase 83.1 already binds.
- Validate and execute the fixed maintained profile: `188` frames, `128` feature bins, and `512`
  encoder-frame width.
- Treat caller-owned workspace as the hot-path scratch surface; do not allocate during execution.

### Integration Contract
- Preserve the existing executor input contract for validated `188 x 512` encoder frames.
- Add a native encoder detail surface that derives those frames from prepared acoustic features.
- Prove composition with a focused test that feeds native encoder output into the existing executor
  without changing module, cache, or transformer expectations.

### the agent's Discretion
- The exact first native pre-encoder numeric slice is at the agent's discretion, but it must use
  maintained encoder/pre-encoder tensors and must not be described as full kernel parity beyond the
  implemented operand path.

</decisions>
