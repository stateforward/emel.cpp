## Implementation Decisions

### User Scope Decision
- The milestone cannot close until full optimized kernel operations are completed.
- Kernel operations belong in the kernel domain.
- Do not maintain two maintained dense/matmul lanes.

### Architectural Correction
- Phase `87` explicitly optimized the Sortformer-local dense helper instead of moving the work into
  `src/emel/kernel/**`. That shortcut is no longer acceptable for milestone closeout.
- Phase `92.3` owns deleting the maintained Sortformer-local dense/matmul path, not preserving it
  as a parallel implementation.

### Kernel Ownership
- Dense/affine/matmul work must dispatch through the existing kernel event surface built around
  `emel::kernel::event::op_mul_mat`.
- Reuse the repo's established kernel request shape from `generator/detail.hpp` and
  `embeddings/generator/detail.hpp` rather than creating a Sortformer-only compute wrapper.
- If the current kernel tensor/view contract is missing something required by Sortformer, extend
  that kernel-owned contract instead of adding another stage-local fallback.

### Phase Boundary
- This phase owns kernel-domain migration and optimized execution for the maintained Sortformer
  dense/matmul hotspots.
- Phase `92.4` owns rerunning profile/benchmark/docs/audit after the kernel migration lands.

</decisions>
