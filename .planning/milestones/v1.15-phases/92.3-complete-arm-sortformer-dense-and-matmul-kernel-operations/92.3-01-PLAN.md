# Phase 92.3 Plan 1: Complete ARM Sortformer Dense And Matmul Kernel Operations

## Goal

Eliminate the maintained Sortformer-local dense/matmul lane by routing the remaining maintained
Sortformer dense work through the kernel domain, then prove the maintained runtime still produces
the truthful pinned output while exposing optimized ARM kernel dispatch on the real maintained
path.

<rule_constraints>
- Do not preserve a second maintained dense/matmul implementation in
  `src/emel/diarization/sortformer/detail.cpp`.
- Kernel operations belong in `src/emel/kernel/**` or another truthful kernel-owned surface.
- Do not add tool-only compute, benchmark-only fast paths, or reference-lane dependencies.
- Do not substitute a dequantize-to-f32 hot-path fallback and present it as final kernel work.
- Keep the maintained lane on the pinned GGUF model and pinned AMI fixture.
- Leave benchmark snapshot/docs republication to Phase `92.4`.
</rule_constraints>

## Tasks

1. Inventory and isolate the duplicate Sortformer dense path.
   - Enumerate every maintained Sortformer callsite that still uses
     `emel::diarization::sortformer::detail::compute_dense(...)`.
   - Confirm which of those sites map directly onto existing kernel `op_mul_mat` semantics and
     which require small kernel-contract extensions.

2. Add kernel-domain wiring for Sortformer dense dispatch.
   - Reuse or extract the tensor/view request construction pattern already used in generator and
     embeddings code.
   - Add any missing kernel-owned helpers needed for Sortformer-owned tensor layouts without
     introducing a Sortformer-only compute lane.

3. Migrate maintained Sortformer stages onto kernel dispatch.
   - Replace Sortformer-local dense calls in encoder, modules, transformer, and output code with
     kernel-domain dispatch.
   - Remove the maintained local dense helper once all maintained callsites are migrated.

4. Revalidate truthful maintained runtime behavior.
   - Prove the maintained request -> pipeline -> segments flow still produces the pinned truthful
     baseline on the real maintained model/audio fixture.
   - Capture kernel-dispatch evidence showing the maintained Sortformer path is now actually using
     the kernel domain for the migrated dense/matmul work.

5. Update phase evidence and state.
   - Write `92.3-01-SUMMARY.md` and `92.3-VERIFICATION.md`.
   - Update `STATE.md` and `ROADMAP.md` to point the next publication/audit step at Phase `92.4`.
