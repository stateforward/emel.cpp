# Phase 92.5 Summary

The existing 92.5 compare-path implementation already had the core ONNX wiring in place:
`tools/bench/diarization_compare.py` can emit both the self-recorded maintained regression lane and
an ONNX reference lane, `tools/bench/diarization_sortformer_onnx_reference.py` runs the ONNX Runtime
CPU path from the maintained exported feature tensor, and
`scripts/bench_diarization_compare.sh` exposes the optional ONNX model input without substituting
the self-recorded regression snapshot when the ONNX lane fails.

This pass tightened the ONNX lane around explicit model provenance and checksum validation:

- `tools/bench/diarization_compare.py` now forwards an expected SHA-256 into the ONNX runner.
- `tools/bench/diarization_sortformer_onnx_reference.py` now validates the ONNX file checksum and
  includes the ONNX path, checksum, and source repository in the emitted note field.
- `scripts/bench_diarization_compare.sh` now documents the expected ONNX SHA-256 and forwards it
  when the ONNX lane is requested.
- `docs/benchmarking.md` and `tests/models/README.md` now record the ONNX model source, expected
  local path, checksum, and runtime expectations.

Focused validation passed:

- `diarization_compare_tests` passed.
- The real maintained compare workflow, executed with a repo-local `build/onnx_ref/venv`
  containing `onnxruntime`, produced:
  - `recorded.diarization.baseline`: exact match
  - `onnx.sortformer.v2_1`: bounded drift

The ONNX bounded drift was the intended scrutiny signal that triggered Phase `92.6`; it is not
papered over by the recorded baseline. The original repo-wide gate timeout was later resolved by
the scoped final quality gate, which completed in `246` seconds with the Sortformer benchmark and
docs generation included.
