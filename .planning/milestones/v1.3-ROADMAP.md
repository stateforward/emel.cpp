## Phase Details

### Phase 14: AArch64 Flash Kernel
**Goal**: Land the backend-specific ARM flash-attention implementation for the maintained
canonical workload contract while preserving reusable workspace semantics.
**Depends on**: v1.2 shipped flash-attention slice
**Requirements**: PORT-01, PORT-02
**Success Criteria** (what must be TRUE):
  1. ARM `op_flash_attn_ext` requests for the maintained canonical contract execute through
     backend-specific optimized code instead of the shared scalar workspace helper.
  2. The optimized ARM path preserves reusable workspace or buffer ownership and does not
     introduce hot-path allocation.
  3. Kernel-facing proof covers canonical correctness and reusable-workspace behavior on AArch64.
**Plans**: 2 plans

Plans:
- [ ] 14-01: Land the AArch64 optimized `op_flash_attn_ext` path for the canonical maintained
  shape and layout contract.
- [ ] 14-02: Prove kernel correctness, reusable workspace behavior, and zero-allocation dispatch
  on the optimized ARM path.

### Phase 15: Runtime Adoption And Proof
**Goal**: Wire the optimized ARM flash path through the shipped runtime chain, keep the existing
Boost.SML orchestration contract intact, and publish explicit proof of supported and unsupported
behavior.
**Depends on**: Phase 14
**Requirements**: PORT-03, ARCH-01, PAR-03, VER-02
**Success Criteria** (what must be TRUE):
  1. The canonical ARM runtime selects the optimized flash path for supported requests and
     publishes explicit deterministic behavior for unsupported or out-of-contract requests.
  2. The shipped execution seam remains the existing generator -> graph -> processor -> kernel
     chain with no queue-based orchestration or API-surface widening.
  3. Parity and regression coverage prove optimized-path execution, negative behavior, and the
     milestone architecture contract on the maintained workload.
**Plans**: 3 plans

Plans:
- [ ] 15-01: Adopt the optimized ARM flash path in the shipped generator -> graph -> processor ->
  kernel chain and publish explicit path-selection evidence.
- [ ] 15-02: Preserve deterministic unsupported or out-of-contract behavior without silent
  optimized-path claims.
- [ ] 15-03: Prove optimized-path execution and negative behavior through parity and regression
  coverage on the maintained canonical ARM workload.

### Phase 16: Benchmark Attribution And Evidence
**Goal**: Publish maintained benchmark evidence that proves and measures the optimized ARM flash
path on the canonical compare workflow.
**Depends on**: Phase 15
**Requirements**: BENCH-04, BENCH-05, BENCH-06
**Success Criteria** (what must be TRUE):
  1. `tools/bench` exercises the optimized ARM flash path on the maintained canonical compare
     workload.
  2. Benchmark outputs distinguish optimized flash execution from surrounding generator/runtime
     cost on the published compare surface.
  3. At least one maintained compare case shows measurable improvement over the current
     shared-scalar ARM flash baseline.
**Plans**: 2 plans

Plans:
- [ ] 16-01: Run the canonical ARM benchmark workload through the optimized flash path on the
  maintained compare surface and publish proof comments or counters.
- [ ] 16-02: Publish benchmark attribution and compare evidence showing the optimization effect
  over the current shared-scalar ARM flash baseline.
