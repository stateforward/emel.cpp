# Milestone v1.0 Requirements: EMEL Llama-68M Generation Slice

**Status:** ✅ SHIPPED 2026-03-08
**Audit:** passed
**Source milestone:** v1.0

## Final Outcome

v1.0 shipped all 14 planned requirements for the narrow Llama-68M paritychecker slice. The only
late audit issue was `HARN-02`, which initially over-claimed fixture specificity; Phase 6 closed
that gap by replacing basename-only acceptance with a canonical path contract.

## v1 Requirements

### Harness

- [x] **HARN-01**: User can invoke a `generation` paritychecker mode with explicit model and prompt
  inputs through `tools/paritychecker/parity_main.cpp`.
- [x] **HARN-02**: User can run the first slice against
  `tests/models/Llama-68M-Chat-v1-Q2_K.gguf` without ambiguous fixture selection.

### GGUF Backend

- [x] **GGUF-01**: User can drive `src/emel/gguf/loader/sm.hpp` probe, bind, and parse events
  against a real GGUF file image so caller-provided storage is populated deterministically for the
  pinned first-slice model.
- [x] **GGUF-02**: User receives deterministic, explicit error reporting from `gguf::loader` for
  malformed GGUF input or insufficient bound storage.

### Loading

- [x] **LOAD-01**: User can load `tests/models/Llama-68M-Chat-v1-Q2_K.gguf` through the EMEL GGUF
  and model loader actors from the paritychecker generation path.
- [x] **LOAD-02**: User receives deterministic, explicit failure reporting when the generation mode
  is given an invalid model path or the EMEL load path rejects the target model.

### Initialization

- [x] **INIT-01**: User can initialize the EMEL generator and required child actors from the loaded
  Llama-68M model inside `tools/paritychecker/`.
- [x] **INIT-02**: The generation initialization path preserves the `docs/rules/sml.rules.md`
  RTC actor model, no-queue invariant, and explicit error publication semantics.

### Generation

- [x] **GEN-01**: User can execute one bounded prompt-to-output generation path through
  `src/emel/generator/sm.hpp` for the Llama-68M parity slice.
- [x] **GEN-02**: User can configure deterministic sampling and stop limits for the parity slice so
  repeated runs are comparable.

### Parity

- [x] **PARI-01**: User can compare EMEL generation output against the `llama.cpp` reference
  implementation for the same prompt and settings inside `tools/paritychecker/`.
- [x] **PARI-02**: User can inspect structured mismatch evidence that goes beyond final rendered
  text, such as token count, stop reason, or other deterministic parity signals.

### Verification

- [x] **VER-01**: User can run `paritychecker_tests` to exercise the generation mode through the
  CLI subprocess surface.
- [x] **VER-02**: User can detect at least one expected failure path for generation mode through
  automated paritychecker tests.

## Outcome Notes

- `HARN-02` was initially only partially satisfied because Phase 1 pinned the fixture by basename.
  Phase 6 hardened that to canonical-path enforcement and closed the audit gap.
- `VER-01` landed earlier than originally planned during Phase 4 when the bounded success-path
  subprocess regression was added.
- The milestone stayed within the agreed acceptance boundary: paritychecker only, no new public C
  API example.
