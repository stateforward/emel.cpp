# Roadmap: EMEL v1.21 Quality Gate Selective Runner Optimization

## v1.21 Quality Gate Selective Runner Optimization

## Overview

v1.21 optimizes the mandatory `scripts/quality_gates.sh` entrypoint by using the
parity and benchmark dependency manifests introduced in v1.18 and v1.19. The milestone preserves
the existing validation contract first, then adds conservative impact resolution, selective
runner execution, safe parallel orchestration, and source-backed validation evidence. The work is
an internal quality-gate orchestration milestone; it does not change parity or benchmark semantics.

## Milestone Context

**Goal:** Teach the required quality gate to run only impacted parity and benchmark runners when
manifest evidence is trustworthy, while failing closed to full relevant validation whenever impact
resolution is missing, stale, uncertain, malformed, or failed.

**Source:** GitHub issue #58, "Optimize quality gates after #54 and #55 land"

**Granularity:** standard

**Requirement coverage:** 14/14 v1 requirements mapped, unmapped 0

## Phases

- [x] **Phase 180: Gate Contract Preservation** - Keep the top-level quality gate mandatory and (completed 2026-05-02)

- [x] **Phase 181: Manifest Impact Resolution** - Derive trustworthy parity and benchmark impact (completed 2026-05-02)
  sets from checked-in dependency manifests and fall back conservatively otherwise.
- [x] **Phase 182: Selective Runner Execution** - Run only selected parity and benchmark runners (completed 2026-05-02)

- [x] **Phase 183: Parallel Orchestration And Reporting** - Run independent lanes concurrently (completed 2026-05-02)

- [x] **Phase 184: Validation Evidence And Closeout** - Prove selective execution improves wall (completed 2026-05-02)
  time without weakening required validation or source-backed closeout evidence.

## Phase Details

### Phase 180: Gate Contract Preservation
**Goal**: Maintainers can keep `scripts/quality_gates.sh` as the mandatory validation contract
while preserving required lane enforcement before selective runner changes land.
**Depends on**: Phase 179 archived in v1.20
**Requirements**: GATE-01, GATE-03
**Success Criteria** (what must be TRUE):
  1. Maintainer can continue to run `scripts/quality_gates.sh` as the required command for
     implementation changes.
  2. Coverage, lint snapshot, docs, fuzz smoke, parity, benchmark, and lane-isolation checks remain
     enforceable unless existing changed-file policy proves a lane irrelevant.
  3. The gate fails visibly instead of silently passing when a required tool, required lane, or
     required validation contract is unavailable.
**Plans**: TBD

### Phase 181: Manifest Impact Resolution
**Goal**: Maintainers can resolve affected parity and benchmark runners from trustworthy
dependency manifests, with conservative full relevant fallback for every uncertain state.
**Depends on**: Phase 180
**Requirements**: IMPACT-01, IMPACT-02, IMPACT-03
**Success Criteria** (what must be TRUE):
  1. Maintainer can provide changed-file inputs and get an impacted parity runner set from present,
     fresh, trustworthy `parity_dependency_manifest/v1` data.
  2. Maintainer can provide changed-file inputs and get an impacted benchmark runner set from
     present, fresh, trustworthy `bench_dependency_manifest/v1` data.
  3. Missing, stale, uncertain, malformed, or failed manifest resolution selects the affected
     tool's full relevant runner set instead of skipping work.
  4. Impact resolution output names the manifest source, selected runners, and fallback reason
     clearly enough to reproduce the decision.
**Plans**: TBD

### Phase 182: Selective Runner Execution
**Goal**: Maintainers can execute only the impacted parity and benchmark runners through the
maintained tool entrypoints when impact resolution is trustworthy.
**Depends on**: Phase 181
**Requirements**: RUNNER-01, RUNNER-02, RUNNER-03
**Success Criteria** (what must be TRUE):
  1. Trustworthy parity impact selects and invokes only the impacted maintained paritychecker
     runners.
  2. Trustworthy benchmark impact selects and invokes only the impacted maintained benchmark
     runners.
  3. Selective-runner decisions reproduce from checked-in manifest baselines and changed-file
     inputs without hidden state or tool-only scaffolds.
  4. Conservative fallback cases still execute through the same maintained paritychecker and
     benchmark entrypoints.
**Plans**: TBD

### Phase 183: Parallel Orchestration And Reporting
**Goal**: Maintainers can run independent quality-gate lanes in parallel without artifact
conflicts, unclear logs, or ambiguous exit status.
**Depends on**: Phase 182
**Requirements**: GATE-02, PAR-01, PAR-02
**Success Criteria** (what must be TRUE):
  1. Independent lanes run concurrently only when their build, test, runtime, and snapshot
     artifacts cannot conflict.
  2. Gate output shows which major lanes ran, which parity or benchmark runners were selected or
     skipped, why each choice was made, and how long each lane took.
  3. Aggregated exit status identifies failed lanes or runners while preserving lane-specific logs
     for local and CI debugging.
  4. Parallel execution remains reproducible enough that a maintainer can rerun or isolate the
     failing lane from the reported command and log context.
**Plans**: TBD

### Phase 184: Validation Evidence And Closeout
**Goal**: Maintainers can close v1.21 only after focused regressions and source-backed evidence
prove faster quality gates without weaker validation.
**Depends on**: Phase 183
**Requirements**: VAL-01, VAL-02, VAL-03
**Success Criteria** (what must be TRUE):
  1. Regression coverage exercises impact resolution, conservative fallback, selective parity
     execution, selective benchmark execution, and parallel result reporting.
  2. End-to-end quality-gate evidence for representative localized changes shows materially
     improved wall time while required validation lanes remain enforceable.
  3. Source-backed audit evidence traces all 14 active requirements to scripts, maintained tool
     entrypoints, tests, and quality-gate output rather than planning artifacts alone.
  4. The milestone closes only after coverage, lint, docs, fuzz, parity, benchmark, and
     lane-isolation guarantees remain intact under the optimized gate.
