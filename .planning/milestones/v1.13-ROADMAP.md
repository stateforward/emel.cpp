generation workloads. It stays bounded to explicit compare contracts, deterministic workload

## Phase Details

### Phase 69: Generative Compare Contract
**Goal**: Define one explicit pluggable compare contract for generative workloads while preserving
strict isolation between the EMEL lane and all reference-engine runtime state.
**Depends on**: None
**Requirements**: GEN-01, GEN-02, ISO-01
**Success Criteria** (what must be TRUE):
  1. The repo has one canonical `generation_compare/v1` contract shared by EMEL and reference
     lanes.
  2. Backend selection stays in manifest/tooling space instead of changing the EMEL runtime lane.
  3. Isolation rules explicitly prevent shared model, tokenizer, sampler, cache, or runtime
     objects between EMEL and any reference backend.
**Plans**: 69-01

### Phase 70: Reproducible Generation Workload Contract
**Goal**: Make maintained generation workloads reproducible and comparable by pinning prompts,
formatter mode, seeds, sampling, and verdict metadata in explicit manifests and records.
**Depends on**: Phase 69
**Requirements**: WRK-01, WRK-02, WRK-03
**Success Criteria** (what must be TRUE):
  1. Compare workloads are driven by explicit manifests that pin model identity, prompt fixture,
     formatter mode, seed, and sampling configuration.
  2. Output records preserve enough metadata to replay a run on the same engine and explain drift
     across engines.
  3. The workflow marks or rejects materially non-comparable runs instead of presenting them as
     equivalent parity results.
**Plans**: 70-01

### Phase 71: Maintained Reference Backend Integration
**Goal**: Bring at least one maintained non-EMEL generative backend onto the shared compare
contract without leaking backend-specific setup into `src/`.
**Depends on**: Phase 70
**Requirements**: REF-01, REF-02, REF-03
**Success Criteria** (what must be TRUE):
  1. At least one maintained generation reference backend runs through the shared compare
     contract on the canonical slice.
  2. Backend-specific setup and environment handling stay confined to the reference lane or its
     wrapper/tooling surface.
  3. Backend failures surface explicit reproducible errors without corrupting the EMEL lane or the
     final compare summary.
**Plans**: 71-01

### Phase 72: Unified Generative Compare Workflow And Publication
**Goal**: Publish one operator-facing EMEL-vs-reference generative compare workflow with truthful
verdicts and machine-readable reproducibility artifacts.
**Depends on**: Phase 71
**Requirements**: CMP-01, CMP-02, CMP-03
**Success Criteria** (what must be TRUE):
  1. Operators can run one consistent compare entrypoint regardless of the selected backend.
  2. Published artifacts include backend identity, workload manifest identity, output summaries,
     and machine-readable compare verdicts.
  3. Compare publication distinguishes exact-match, bounded-drift, and non-comparable outcomes
     explicitly.
**Plans**: 72-01

### Phase 73: Proof, Regression, And Milestone Closeout
**Goal**: Close the milestone with maintained end-to-end regression coverage, explicit requirement
traceability, and audit-ready proof for the approved generative workload boundary.
**Depends on**: Phase 72
**Requirements**: PRF-01, PRF-02
**Success Criteria** (what must be TRUE):
  1. Repo-owned regression coverage reproduces at least one maintained multi-engine generative
     compare path end to end.
  2. Stored milestone evidence documents the approved workload boundary and any remaining
     comparability caveats truthfully.
  3. Requirement traceability, validation, and closeout artifacts are complete enough for a clean
     milestone audit.
**Plans**: 73-01

### Phase 74: Generation Compare Lane Isolation Repair
**Goal**: Close the `v1.13` audit blocker where EMEL JSONL compare output enters compare fixture
setup that prepares reference runtime state.
**Depends on**: Phase 73
**Requirements**: GEN-01, ISO-01, REF-02
**Gap Closure**: Closes `v1.13` milestone audit gaps for EMEL/reference lane isolation.
**Success Criteria** (what must be TRUE):
  1. EMEL JSONL output never prepares, probes, initializes, or otherwise touches reference-engine
     model, tokenizer, sampler, cache, or runtime objects.
  2. Reference backend selection stays confined to the reference wrapper/tooling lane and cannot
     alter the EMEL generation lane implementation.
  3. Regression coverage proves the EMEL JSONL lane remains isolated while comparable reference
     runs still work through the operator compare workflow.
**Plans**: 74-01

### Phase 75: Comparability Verdict And Single-Lane Publication Repair
**Goal**: Close the `v1.13` audit blockers around real non-comparable publication and incomplete
comparability verdict inputs.
**Depends on**: Phase 74
**Requirements**: WRK-03, CMP-03, PRF-01
**Gap Closure**: Closes `v1.13` milestone audit gaps for single-lane publication and verdict
truthfulness.
**Success Criteria** (what must be TRUE):
  1. A maintained single-lane workload can flow through the operator-facing compare workflow and
     publish a truthful `non_comparable` summary without synthetic JSONL-only scaffolding.
  2. Compare verdict logic consumes all material comparability fields before publishing
     `exact_match`, `bounded_drift`, or `non_comparable` outcomes.
  3. Regression coverage proves both the comparable multi-engine workflow and real single-lane
     non-comparable workflow end to end.
**Plans**: 75-01
