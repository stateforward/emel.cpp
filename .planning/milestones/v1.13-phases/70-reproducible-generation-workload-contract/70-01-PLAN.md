# Phase 70 Plan 01

<rule_constraints>
- Do not widen `src/emel/generator` runtime scope or move compare-workload control into runtime
  actors.
- Keep the maintained compare path on checked-in repo-owned manifests and prompt fixtures.
- Mark non-comparable workloads explicitly instead of allowing silent parity claims.
</rule_constraints>

## Objective

Make maintained generation workloads reproducible and comparable by loading prompt and workload
identity from explicit manifests and carrying that provenance through `generation_compare/v1`
records.

## Tasks

1. Add checked-in generation prompt fixtures and workload manifests that pin fixture identity,
   prompt identity, formatter contract, sampling, stop, seed, max tokens, and comparability.
2. Update the maintained generation bench to load those manifests, emit their provenance in
   `generation_compare/v1`, and keep compare mode restricted to comparable workloads.
3. Add focused regression coverage and docs proving the manifest-driven contract and the explicit
   non-comparable labeling path.

## Verification

- `cmake --build build/bench_tools_ninja --parallel --target bench_runner bench_runner_tests`
- `ctest --test-dir build/bench_tools_ninja --output-on-failure -R bench_runner_tests`
- `./build/bench_tools_ninja/bench_runner_tests --test-case="bench_runner generation jsonl emits manifest-driven workload metadata and explicit comparability"`

## Done Condition

Phase 70 is complete when maintained generation workloads are loaded from checked-in manifests,
`generation_compare/v1` records expose replayable workload provenance, and non-comparable
single-lane workloads are marked explicitly rather than being reported as parity.
