# Phase 74 Plan: Generation Compare Lane Isolation Repair

<rule_constraints>
## Applicable Rules

- EMEL and reference benchmark lanes must stay separated; no shared model, tokenizer, sampler,
  cache, or runtime objects in the EMEL lane.
- Reference-engine setup must stay confined to the reference lane or wrapper/tooling surface.
- The repair must preserve existing compare output behavior and avoid fallback runtime scope creep.
</rule_constraints>

## Goal

Restore strict EMEL/reference separation for generation JSONL output by ensuring `--mode=emel`
does not enter compare fixture preparation when JSONL output is requested.

## Tasks

1. Add focused regression coverage in `tools/bench/bench_runner_tests.cpp`.
   - Prove EMEL JSONL still emits canonical `generation_compare/v1` EMEL records.
   - Prove reference JSONL still emits canonical reference records.
   - Preserve any existing single-lane visibility checks for hosts with the Gemma4 fixture.

2. Repair CLI lane-mode selection in `tools/bench/bench_main.cpp`.
   - Set `generation_lane_mode::emel` for `--mode=emel` regardless of JSONL output format.
   - Set `generation_lane_mode::reference` for `--mode=reference` regardless of JSONL output
     format.
   - Leave `--mode=compare` on `generation_lane_mode::compare`.

3. Verify the focused surface.
   - Build `bench_runner` and `bench_runner_tests`.
   - Run the generation JSONL bench runner test.
   - Run `generation_compare_tests` to ensure the operator workflow still works.

## Expected Evidence

- `bench_runner_tests` passes for JSONL output.
- `generation_compare_tests` passes.
- Phase verification documents the lane isolation evidence and the remaining Phase 75/76 follow-up
  boundaries.
