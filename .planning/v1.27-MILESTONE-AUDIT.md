# v1.27 Milestone Audit - co_sm Cooperative Async I/O Strategy

## Result

**Status: gaps_found.** The milestone has complete phase-level verification files and the
three-source requirement cross-reference covers all 31 active requirements, but cross-phase source
inspection found two blockers that should be closed before treating the milestone as clean:

1. `AIO-06` claims scheduler/resource terminal error semantics, but the invalid scheduler route in
   `src/emel/io/async/guards.hpp` is unreachable at runtime.
2. `DOC-01` / `EVI-01` claim truthful milestone evidence, while `ROADMAP.md` still contains a stale
   progress table marking phases 241-247 as not started.

The maintained `cooperative_async` generation path does execute through public `io/loader` and
`emel::IoAsync` contracts, and Phase 248 records benchmark evidence. That satisfies the core
maintained-path proof, with one caveat: the loader drains all async chunks internally before
returning to model-loader/generation callers.

## Integration Check

The integration checker found the main maintained path wired:

`EMEL_MODEL_LOAD_IO_STRATEGY=cooperative_async` -> `model_load_strategy.hpp` ->
`generation_bench.cpp` -> `model/loader` -> `io/loader` -> `emel::IoAsync`.

Connected public contracts:

- `emel::co_sm` is consumed by `src/emel/io/async/sm.hpp` and covered by public wrapper tests.
- `io::async::event::load_window` contracts are consumed by `io/async`, `io/loader`, and direct
  `model/tensor` async actions.
- `io::loader::event::strategy_kind::cooperative_async` is consumed by loader guards/transitions,
  model-loader guards, benchmark strategy parsing, and maintained generation setup.
- `emel::IoAsync` is consumed by `tools/bench/generation_bench.cpp` through the public alias.

Integration gaps and caveats:

| ID | Requirements | Status | Evidence |

| INT-02 | AIO-06 | blocker | Scheduler-contract error routing is modeled but unreachable because valid/invalid scheduler guards are constant true/false. |
