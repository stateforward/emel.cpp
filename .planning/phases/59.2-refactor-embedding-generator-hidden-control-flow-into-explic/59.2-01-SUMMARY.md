# Phase 59.2 Summary

## Outcome

Phase 59.2 is complete. The maintained `src/emel/embeddings/generator` actor now makes the
scoped prepare and encode route decisions explicit in guards and transitions instead of routing
through detail-side bool-to-error wrapper helpers. The model stays on the public
`process_event(...)` path, the guard-visible route structure is cleaner, and the old
helper-latched prepare/encode seams are gone.

## Delivered

- Added an explicit source-inspection regression in
  `tests/embeddings/generator_action_guard_tests.cpp` to lock out the old prepare/encode latch
  pattern, the detail-side `*_error` wrapper helpers, and the old post-action backend/success
  latch guards.
- Refactored the maintained generator actions so the route-specific prepare/encode actions call
  the chosen data-plane helper directly instead of routing through `detail.hpp` error-wrapper
  helpers.
- Added explicit route-readiness guards for the maintained text, image, and audio encode/prepare
  lanes so the machine publishes backend error before the action runs when a supported route is
  not execution-ready.
- Simplified the maintained state machine by removing the old image/audio prepare-decision latch
  seam and the old encode-publication backend latch seam.
- Removed the now-unused detail helpers:
  - `prepare_image_input_mobilenetv4_error(...)`
  - `prepare_audio_input_efficientat_error(...)`
  - `run_text_embedding_bert_error(...)`
  - `run_image_embedding_mobilenetv4_error(...)`
  - `run_audio_embedding_efficientat_error(...)`

## Resulting Model Shape

- Route choice remains explicit through:
  - `guard_text_route_bert`
  - `guard_image_route_mobilenetv4`
  - `guard_audio_route_efficientat`
- Route readiness is now explicit through:
  - `guard_text_encode_bert_ready` / `_unready`
  - `guard_image_prepare_mobilenetv4_ready` / `_unready`
  - `guard_image_encode_mobilenetv4_ready` / `_unready`
  - `guard_audio_prepare_efficientat_ready` / `_unready`
  - `guard_audio_encode_efficientat_ready` / `_unready`
- Publish choice is now only the full-vs-truncate decision after the already-chosen encode path
  runs; it no longer routes through a helper-driven backend latch.

## Notes

- This phase removed the ad hoc prepare/encode outcome wrapper seam and the corresponding
  post-action latch routing from the maintained actor. It did not try to re-model every
  data-plane branch inside the numeric image/audio/text kernels as state-machine structure; the
  phase stayed scoped to the actor-level hidden control-flow seams called out by the user.
- The benchmark snapshot compare still produced the existing warning-tolerated regression note:
  `logits/sampler_sml/vocab_256000`.
