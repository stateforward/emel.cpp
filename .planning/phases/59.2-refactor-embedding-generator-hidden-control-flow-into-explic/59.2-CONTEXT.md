# Phase 59.2 Context

## Phase Boundary

Phase 59.2 exists because the maintained `embeddings/generator` actor still hides runtime behavior
and outcome selection in `actions.hpp` / `detail.hpp`, which conflicts with the repo SML rules and
the user’s explicit direction to make the model behavior visible through guards rather than hidden
helper logic. This phase stays on the maintained TE embedding-generator path in `src/`. It does
not widen modality scope, and it does not replace the runtime with tool-only scaffolding or ad hoc
fallbacks.

## Implementation Decisions

### Scope
- Refactor only the maintained `src/emel/embeddings/generator` model and its directly owned
  runtime helpers.
- Remove remaining hidden runtime outcome selection from prepare and encode paths.
- Remove remaining hidden runtime behavior selection from maintained generator helpers where the
  helper is choosing which path runs, not merely executing data-plane numeric work.
- Keep all request handling on the public `process_event(...)` path of the owning machine.

### Constraints
- No helper-local ad hoc dispatch tables, fallback arrays, or workaround routing.
- No new public API surface.
- No tool-only compute or reference-side shortcuts.
- No dispatch-time allocation.
- Any state-machine structural change is allowed in this phase because the user explicitly asked
  for the model to be made explicit through guards.

### Rule Intent
- Runtime success/error choice must be visible in guards and transitions, not buried behind bool
  returns from `detail.hpp`.
- Runtime behavior choice must be visible in the model, not hidden in generic helper routing.
- Data-plane numeric iteration remains allowed in `detail.hpp`; this phase is about orchestration
  and behavior choice, not banning every branch in numeric kernels.

## Existing Code Insights

### Current Hidden Outcome Selection
- `effect_prepare_image_input` and `effect_prepare_audio_input` currently set `ev.ctx.prepared`
  from bool-returning `detail::prepare_*` helpers, and guards route on that result.
- `effect_run_text_embedding`, `effect_run_image_embedding`, and `effect_run_audio_embedding`
  currently set `ev.ctx.embedded` from bool-returning `detail::run_*` helpers, and guards route on
  that result.

### Current Hidden Behavior Selection
- `detail::matmul(...)` still selects runtime dtype routes inside `detail.hpp`.
- `run_universal_inverted_block(...)` and `run_audio_block(...)` still select behavior from model
  block flags inside `detail.hpp`.
- `run_image_embedding(...)` and `run_audio_embedding(...)` still rely on generic bool-returning
  helpers to determine whether the actor publishes success or backend error.

### Prior Phase Boundary
- Phase 57 removed explicit `if` statements from the maintained action bodies, but it intentionally
  stopped short of removing the remaining bool-based routing and helper-hidden behavior selection
  now called out by the user.

## Specific Ideas

- Add a focused regression that inspects the maintained embedding-generator source for the
  remaining hidden control-flow seams called out in this phase.
- Replace bool-latched prepare/encode results with explicit guard-visible phase outcomes.
- Move route selection out of generic helper dispatch where the helper is deciding behavior rather
  than executing already-chosen numeric work.
- Keep the final runtime shape truthful: no ad hoc “temporary” fallback lanes just to satisfy the
  rule audit.

## Deferred Ideas

- Further image depthwise optimization
- Additional ARM throughput work
- Any new modality or model-family scope
