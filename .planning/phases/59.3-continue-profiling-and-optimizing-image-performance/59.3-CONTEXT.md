# Phase 59.3 Context

## Phase Boundary

Phase 59.3 exists because the maintained single-thread image request path is still too slow for the
edge target, and the latest post-`59.2` sample changed the ROI ordering again. This phase stays on
the maintained `src/emel/embeddings/generator` image request path and its maintained steady-state
benchmark surface. It does not widen scope to new models, multi-threading, or reference-lane
compute.

## Implementation Decisions

### Scope
- Measure only the maintained image request path through
  `tools/bench/embedding_generator_bench.cpp`.
- Keep throughput steady-state only; `initialize(...)` remains excluded.
- Treat the maintained image `prepare` / `encode` / `publish` split as the source of truth.
- Optimize only the image pointwise seam in this phase; depthwise and audio stay out of scope
  unless the updated profile after the pointwise pass makes a different next ROI explicit.

### ROI Order
- Attack the current image hotspot first: the pointwise lowering around the maintained AArch64 f32
  matmul kernel inside `run_universal_inverted_block(...)`.
- Prefer removing the pack/unpack seam entirely over retuning the current tile constant.
- Rebenchmark and reprofile immediately after the pointwise pass before taking any second image
  optimization.
- Leave depthwise alone in this phase unless the new profile proves it moved materially up the
  stack.

### Constraints
- Keep all compute in EMEL-owned `src/` code.
- No helper-local benchmark shortcuts or reference-side compute.
- No hidden runtime control-flow selection in actions/detail helpers.
- Any new storage must be one-time bind-time or reserve-time memory only; no dispatch-time
  allocation.
- Keep the image tower on the owning machine public event path.

## Existing Code Insights

### Current Maintained Image Evidence
- Fresh isolated maintained image result for this phase:
  - `237262162.500 ns/op` (`~237.262 ms`, `~4.21 embeddings/s`)
  - `prepare=4779808.300 ns`
  - `encode=232461920.800 ns`
  - `publish=1062.500 ns`
- Fresh 5-second sample on the maintained image-only run shows:
  - `run_universal_inverted_block(...)` still dominating the image lane
  - `pointwise_conv_hwc_tiled_f32(...)` dominating inside those blocks
  - `execute_neon_mul_mat(...)` dominating inside that pointwise path
- That fresh sample no longer supports the earlier depthwise-first assumption as the next ROI.

### Current Lowering Reality
- The maintained image pointwise path still expands weights once, then pays a per-request lowering
  seam:
  - pack HWC input tile into a dense matrix
  - run f32 matmul
  - unpack dense output back to HWC
- The current pointwise helper also hard-caps tile size at `64` pixels even when the one-time
  scratch capacity is much larger.
- The maintained AArch64 f32 matmul kernel already supports dense f32 matrix work efficiently when
  operands are shaped the way it expects.

### Practical Implementation Bias
- The cleanest high-ROI move is to bind pointwise weights once in the transposed dense layout that
  lets the kernel consume image pixels directly as the left-hand matrix and write HWC output
  directly as the destination matrix.
- That should let the maintained image path remove both the input pack and output unpack seam
  around the current hotspot instead of merely moving its tile size.

## Specific Ideas

- Add one-time transposed dense pointwise weights at bind time for the maintained f16 pointwise
- matrices used by the image tower.
- Add a maintained direct HWC pointwise helper that treats the pixel matrix as the left-hand dense
  operand and the transposed pointwise weights as the right-hand dense operand.
- Rewire the image-only pointwise callers (`stage0`, `pw_exp`, `pw_proj`) to that direct path.
- Remove the now-unneeded image pointwise workspace if the new direct path makes it dead.
- Re-sample the image-only benchmark after the first pass to decide whether depthwise or shared
  kernel tuning is the next ROI.

## Deferred Ideas

- Depthwise image kernel work
- Audio frontend optimization
- Text throughput work
- Reference-lane comparison changes
