# Plan 148-01: Parity Runner Asset Boundary

<objective>
Create a shared paritychecker asset boundary for common file, path, byte-loading, and maintained
generation fixture resolution logic while preserving all existing mode behavior and lane ownership.
</objective>

<rule_constraints>
## Rule Traceability

- `LANE-01` requires the EMEL and reference lanes to keep model/vocab/tokenizer/runtime/cache state
  separately owned. Tasks 2 and 3 only move paths, bytes, and fixture lookup helpers; they do not
  share lane runtime objects.
- The reference policy requires maintained fixture claims to be source-backed. Tasks 1 and 4 add a
  testable asset boundary and source/process regressions for the maintained generation fixture
  contract.
- Existing parity behavior must remain stable during the refactor. Tasks 3 and 5 keep mode dispatch
  and per-mode implementations unchanged while adding focused paritychecker tests.
</rule_constraints>

<task id="2" type="auto">
<title>Add parity asset helper boundary</title>
<read_first>
- `tools/generation_fixture_registry.hpp`
- `tools/paritychecker/CMakeLists.txt`
</read_first>
<instructions>
Add `parity_assets.hpp` / `parity_assets.cpp` under namespace `emel::paritychecker::assets`.

<acceptance_criteria>

- The helper carries no EMEL/reference runtime objects and no tokenizer/model/cache ownership.
