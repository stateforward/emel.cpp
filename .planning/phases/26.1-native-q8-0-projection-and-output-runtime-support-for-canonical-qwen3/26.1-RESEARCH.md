## Summary

The confirmed blocker is native `q8_0` execution, not more Qwen-specific planning. The real
maintained parity path currently prints canonical Qwen3 formatter contract details and execution-view
debug information, then fails with `generation initialize failed (fixture=Qwen3-0.6B-Q8_0.gguf
err=backend)`. The reported blocking tensors are `attn_q`, `attn_k`, and tied output projection,
all with GGUF type `8` (`q8_0`).

EMEL already declares `dtype::q8_0` in [src/emel/kernel/events.hpp](/Users/gabrielwillen/VSCode/stateforward/emel/emel.cpp/src/emel/kernel/events.hpp),
but native support is incomplete in the exact places the canonical Qwen3 hot path uses:
- [src/emel/kernel/detail.hpp](/Users/gabrielwillen/VSCode/stateforward/emel/emel.cpp/src/emel/kernel/detail.hpp) has no `block_q8_0`, no `dequantize_row_q8_0`, and no `q8_0 x f32` path in `run_mul_mat` or `run_mul_mat_argmax`.
- [src/emel/generator/detail.hpp](/Users/gabrielwillen/VSCode/stateforward/emel/emel.cpp/src/emel/generator/detail.hpp)
  cannot dequantize a `q8_0` tensor row through `copy_tensor_row`, which matters for tied
  embedding/output access on the canonical slice.
- [src/emel/model/data.cpp](/Users/gabrielwillen/VSCode/stateforward/emel/emel.cpp/src/emel/model/data.cpp)
  currently treats only the `q2_k/q3_k/q6_k` family as native quantized in audit classification,
  which becomes stale once `q8_0` execution is added.

Local reference inspection gives a narrow arithmetic target. In `tmp/llama.cpp`, `q8_0` is a
32-value block with one fp16 scale and 32 signed int8 quants (`block_q8_0` in
`ggml-common.h`). `dequantize_row_q8_0` is a simple `qs[j] * d` expansion, and the generic dot
product path is the blockwise sum of `int8 * int8`, scaled by the product of the block scales
(`ggml_vec_dot_q8_0_q8_0_generic`). That fits cleanly into EMEL's existing scalar kernel structure:
quantize the f32 RHS vector into `q8_0` blocks, dot each `q8_0` matrix row against that temporary
RHS, and keep the whole path inside `src/emel/kernel/detail.hpp`.

**Primary recommendation:** Plan Phase 26.1 as two narrow steps. First, add explicit native `q8_0`
row storage, dequantization, and scalar `op_mul_mat` / `op_mul_mat_argmax` support in
`src/emel/kernel/detail.hpp` plus hot-path `copy_tensor_row` support in
`src/emel/generator/detail.hpp`. Second, prove that the canonical Qwen3 bring-up consumes this new
support and advances past the old `err=backend` failure, while leaving full maintained generation
proof to resumed Phase 27.

## Architecture Patterns

### Pattern 1: Add an explicit `q8_0` family next to existing quantized helpers
**What:** Introduce `QK8_0`, `block_q8_0`, `quantize_row_q8_0`, `dequantize_row_q8_0`, and a
scalar `dot_q8_0_q8_0_row`.
**Why:** EMEL already models q2/q3/q6 helpers explicitly in `kernel/detail.hpp`; `q8_0` should be
added the same way, not hidden behind ad hoc byte math.

### Pattern 2: Keep `q8_0` execution separate from the `quantized_k` family
**What:** Do not pretend `q8_0` is another `QK_K=256` family member. Give it its own explicit
predicates and storage sizing.
**Why:** Reference `q8_0` uses 32-value blocks, not 256-value K-blocks. Reusing the wrong family
would produce invalid row-byte checks and misleading capability claims.

### Pattern 3: Let generator helpers consume native support, not replace it
**What:** Extend `copy_tensor_row` and matrix binding validation to admit `q8_0`, but keep actual
numeric execution in kernel helpers.
**Why:** This phase should unblock runtime use of existing Qwen3 groundwork rather than invent a
second compute path in generator.

## Common Pitfalls

### Pitfall 1: Treating `q8_0` as `q8_k`
**What goes wrong:** Validation accepts the tensor type, but row storage and dot-product math are
wrong because EMEL uses the 256-value `q8_k` helper layout.
**Avoid by:** Adding a distinct `block_q8_0` path and explicit row-byte calculation.

### Pitfall 2: Solving the blocker with row dequantization only
**What goes wrong:** Token embedding reads work, but projection/output matrix execution still fails
because `op_mul_mat` and `op_mul_mat_argmax` remain unsupported.
**Avoid by:** Treat row decode and matrix/vector execution as one narrow support slice.

### Pitfall 3: Broadening audit/output claims before parity is resumed
**What goes wrong:** Phase 26.1 starts sounding like full Qwen parity support instead of a runtime
prerequisite.
**Avoid by:** Limit verification to native `q8_0` support and explicit backend-unblock proof; keep
full maintained generation proof in Phase 27.
