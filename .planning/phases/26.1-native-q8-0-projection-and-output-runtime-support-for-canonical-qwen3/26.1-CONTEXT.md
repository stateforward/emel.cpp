# Phase 26.1: Native q8_0 Projection And Output Runtime Support For Canonical Qwen3 - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 26.1 exists because the blocked `27-02` attempt proved that canonical Qwen3 runtime/parity
bring-up cannot finish until EMEL-owned code in `src/emel` executes the fixture's `q8_0`
projection/output tensors natively. This phase is therefore a prerequisite recovery phase, not a
tooling workaround.

The phase must stay narrow to the hot-path runtime support needed to unblock the canonical
`Qwen3-0.6B-Q8_0.gguf` slice. It must not widen to broad `q8_0` model support claims, benchmark
publication, or tool-only compute fallbacks. The existing Boost.SML actor structure stays intact.

</domain>

<decisions>
## Implementation Decisions

### Native Runtime Support Only
- The missing support must land in EMEL-owned `src/` runtime/kernel code, not in
  `tools/paritychecker` or other tool-local compute fallbacks.
- Do not dequantize entire projection/output tensors to f32 as a hot-path substitute. The user has
  not approved that fallback, and repo rules explicitly forbid presenting that as done.
- Keep the support explicit and modeled: add only the runtime/kernel paths actually required by the
  canonical Qwen3 slice.

### Narrow Support Contract
- Scope the work to the native operations currently blocking canonical Qwen3 generation:
  `q8_0` matrix/vector execution for the shipped generator hot path and any required `q8_0` row
  reads used by that path.
- Treat the current `27-02` partial changes as downstream consumers of this phase, not as a reason
  to broaden Phase 26.1 into generic Qwen bring-up.
- Preserve the truthful milestone contract: this phase unblocks runtime capability; parity and
  broader regression/publication remain later phases.

### Architecture And Control-Flow Guardrails
- Keep the current generator -> graph -> processor -> kernel actor chain and public API shape.
- Do not change state machine structure for this phase; stay inside helper/runtime/kernel paths.
- Keep all runtime behavior explicit. No hidden fallback path, no model-name heuristic branching
  beyond the already-approved canonical Qwen3 architecture handling.

### the agent's Discretion
- The exact split between generator helper changes and kernel detail changes can stay local as long
  as the final support is native, narrow, and grep/test verifiable.

</decisions>

<code_context>
## Existing Code Insights

### Verified Blocker Evidence
- The real maintained parity command currently fails after the partial `27-02` updates with:
  `generation initialize failed (fixture=Qwen3-0.6B-Q8_0.gguf err=backend)`.
- The current debug output shows the canonical Qwen3 hot-path tensors are `type=8` (`q8_0`) for
  at least:
  - `attn_q` (`1024x2048`)
  - `attn_k` (`1024x1024`)
  - `output` (`1024x151936`)
- The same debug output shows `output_weight=0`, so the tied-output fallback path is already part
  of the real canonical slice and must work natively when it binds a `q8_0` matrix.

### Relevant Runtime Gaps
- `src/emel/kernel/events.hpp` declares `dtype::q8_0`, but `src/emel/kernel/detail.hpp` does not
  implement a native `q8_0` execution path for `op_mul_mat` or `op_mul_mat_argmax`.
- `src/emel/kernel/detail.hpp` currently treats only `q2_k`, `q3_k`, and `q6_k` as quantized-k
  matrix paths in the generic scalar backend.
- `src/emel/generator/detail.hpp` currently:
  - binds projection/output matrices generically through `bind_tensor_rows`
  - uses kernel `op_mul_mat` / `op_mul_mat_argmax` for runtime matrix-vector execution
  - only has logits packing/preparation fast paths for `q6_k`
  - does not support `copy_tensor_row` for `q8_0`, which matters for tied token embedding/output
    row access on the canonical Qwen3 slice

### Reusable Assets
- `src/emel/generator/detail.hpp` already has the surrounding runtime structure for prefill, decode,
  logits, and the approved Qwen3 Q/K norm ordering.
- `src/emel/kernel/detail.hpp` already owns the native scalar/reference semantics for supported
  quantized matrix paths, so this phase can extend the existing kernel detail surface instead of
  inventing a new tool-only path.
- `tests/generator/detail_tests.cpp`, `tests/generator/lifecycle_tests.cpp`, and
  `tools/paritychecker/paritychecker_tests.cpp` already contain the new fail-first Qwen3 generator
  coverage committed in `2bc89b3`.

### Existing Partial Work
- The paused `27-02` workspace already contains uncommitted downstream changes in:
  - `src/emel/generator/detail.hpp`
  - `tools/paritychecker/parity_runner.cpp`
  - `tools/paritychecker/paritychecker_tests.cpp`
  - `tests/generator/detail_tests.cpp`
- Those changes prove the remaining blocker is native backend support, but they are not yet phase
  artifacts and should not be treated as complete work.

</code_context>

<specifics>
## Specific Ideas

- The narrowest truthful implementation is likely:
  1. add native `q8_0` row decode support needed by generator hot-path reads,
  2. add native `q8_0 x f32` matmul support for `op_mul_mat`,
  3. add native `q8_0 x f32` argmax support for `op_mul_mat_argmax`,
  4. then let the existing partial `27-02` Qwen runtime/parity wiring consume that support.
- Because the canonical Qwen3 fixture exposes `attn_q`, `attn_k`, and tied output projection as
  `q8_0`, Phase 26.1 should verify those exact tensor classes rather than claiming generic
  `q8_0` coverage everywhere.
- If any helper needs to distinguish supported quantized classes, the distinction must be explicit
  in code and tests, not inferred from tool output.

</specifics>

<deferred>
## Deferred Ideas

- Broad `q8_0` support claims beyond the operations needed for canonical Qwen3 generation.
- Benchmark/publication work.
- Generic multi-quant or broader Qwen-family runtime widening.
- Any whole-tensor dequantize-to-f32 fallback in the hot path.

</deferred>
