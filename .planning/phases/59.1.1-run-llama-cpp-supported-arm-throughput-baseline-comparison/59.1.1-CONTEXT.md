# Phase 59.1.1 Context

## Phase Boundary

Phase 59.1.1 exists to replace internal-only ARM throughput claims with a same-host external
baseline matrix. The agreed reference set is fixed to `Arctic S`, `EmbeddingGemma 300M`,
`LFM2-VL-450M`, and `Ultravox 1B`. This phase stays inside `tools/bench`, `scripts/`, and
phase/bookkeeping artifacts; it does not widen the EMEL runtime scope beyond the maintained text,
image, and audio embedding-generator path.

## Implementation Decisions

### Scope
- Keep the benchmark lanes strictly separated:
  - EMEL lane uses only EMEL-owned code and the maintained `embeddings/generator` benchmark.
  - Reference lane uses only `llama.cpp` / `mtmd` APIs and its own model assets.
- Measure steady-state throughput only; initialization and model load do not count toward reported
  embeddings/s.
- Preserve the existing `prepare` / `encode` / `publish` framing. When a reference lane cannot
  expose an exact stage match, publish the closest truthful attribution and call out the mismatch
  explicitly.
- Use the same canonical payloads already locked for the TE lane:
  - text: `red square`
  - image: `32x32` red square
  - audio: mono `f32` `16 kHz` `440 Hz` tone

### Current Gap
- The current reference runner only covers Liquid `mtmd` image/audio and skips cleanly when
  multimodal assets are missing.
- There is no text-reference lane yet for `Snowflake/snowflake-arctic-embed-s` or
  `EmbeddingGemma 300M`.
- The phase is not done until the same-host numbers are actually produced or a hard external asset
  blocker is proven concretely.

### Asset Strategy
- Prefer official or official-organization GGUF artifacts that can be loaded directly by the
  reference benchmark.
- Store maintained comparison fixtures under `tests/models/` with provenance added to
  `tests/models/README.md` when new local assets are committed into the maintained surface.
- Keep multimodal projector/model assets configured by explicit environment variables when they are
  too large or too variant-specific to treat as pinned maintained fixtures immediately.

### Constraints
- Do not let the EMEL benchmark lane call into reference helpers or share model/tokenizer/context
  objects with the reference lane.
- Do not use helper-local shortcuts for the EMEL numbers.
- Do not claim apples-to-apples parity where the reference lane lacks an exact stage split.

## Existing Code Insights

### Existing EMEL Evidence
- The maintained EMEL ARM numbers already live in
  `tools/bench/embedding_generator_bench.cpp` and Phase `59.1`.
- Those numbers already exclude `initialize(...)` and publish `prepare`, `encode`, and `publish`
  timing for text, image, and audio.

### Existing Reference Surface
- `tools/bench/embedding_reference_bench.cpp` currently uses `libmtmd` directly for image/audio.
- `scripts/bench_embedding_reference_liquid.sh` builds a dedicated reference runner against the
  fetched reference repo.
- The current reference runner lacks text embedding cases and model-matrix metadata for the agreed
  text anchors.

### Reference API Reality
- The fetched reference source already exposes a direct embedding API through `llama_decode`,
  `llama_get_embeddings_seq`, and the standard tokenizer APIs for text embedding models.
- The fetched reference source also exposes `mtmd_tokenize`, `mtmd_encode_chunk`, and
  `mtmd_get_output_embd` for multimodal image/audio inputs.

## Specific Ideas

- Add a text reference path to `embedding_reference_bench_runner` that can measure `Arctic S` and
  `EmbeddingGemma 300M` without shelling out to `llama-embedding`.
- Generalize the reference runner metadata so each case records which model lane produced it.
- Add a maintained shell entrypoint that can run the full matrix or filtered subsets on the same
  host.
- If the multimodal assets are not present locally, bootstrap them explicitly and record the exact
  environment contract used for the benchmark run.

## Deferred Ideas

- Broader video benchmarking
- Publishing parity claims beyond throughput/stage anchors
- Pulling the reference lane into `src/` or exposing it through public EMEL APIs
