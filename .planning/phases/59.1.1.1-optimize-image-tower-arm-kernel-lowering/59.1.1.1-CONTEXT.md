# Phase 59.1.1.1 Context

## Phase Boundary

Phase 59.1.1.1 exists because the maintained single-thread image path is still far below the
needed ARM throughput for an edge model, and the latest profile makes the ROI order explicit. This
phase stays on the maintained `src/emel/embeddings/generator` image request path and its
maintained single-thread benchmark surface. It does not widen modality scope, and it does not
replace the image tower with tool-only or reference-side compute.

## Implementation Decisions

### Scope
- Measure only the maintained single-thread image request path through
  `tools/bench/embedding_generator_bench.cpp`.
- Keep throughput steady-state only; `initialize(...)` remains excluded.
- Treat the maintained image `prepare` / `encode` / `publish` split as the source of truth.
- Optimize only the image tower in this phase; audio and text are out of scope unless a shared
  kernel-side cleanup falls out of the chosen image fix.

### ROI Order
- Attack universal inverted-block pointwise `1x1` lowering first.
- Rebenchmark and reprofile immediately after that first pass.
- Only take the edge-residual spatial conv or shared f32 packing cleanup in this phase if the
  updated profile still shows them as the next highest-ROI follow-on.
- Leave depthwise alone unless the new profile proves it moved materially up the stack.

### Constraints
- Keep all compute in EMEL-owned `src/` code.
- No helper-local benchmark shortcuts or reference-lane compute.
- No hidden runtime control-flow selection in actions/detail helpers.
- Any new workspace must be one-time initialized scratch only; no dispatch-time allocation.
- Keep the image tower on the owning machine public event path.

## Existing Code Insights

### Current Maintained Image Evidence
- Latest isolated maintained ARM result from Phase `59.1`:
  - Image: `~1.51 embeddings/s` (`661.395 ms`)
- Latest focused profile on the maintained single-thread image benchmark:
  - `~95%` top-of-stack in `execute_neon_mul_mat`
  - `~80%` inclusive in `run_universal_inverted_block`
  - `~16%` inclusive in `run_edge_residual`
  - preprocessing is negligible

### Current Image-Tower Lowering Reality
- `pointwise_conv_hwc(...)` in `src/emel/embeddings/generator/detail.hpp` still loops pixels and
  calls generic `matmul(...)` once per pixel.
- `run_universal_inverted_block(...)` calls that path twice per block (`pw_exp` and `pw_proj`).
- `run_edge_residual(...)` still pays one `standard_conv_hwc(...)` plus one pointwise pass.
- Image pointwise weights are already expanded once at bind time to dense f32, so the missing win
  is the lowering shape, not repeated fp16 decode.

### Practical Implementation Bias
- The maintained AArch64 kernel already supports matrix-matrix f32 `op_mul_mat`.
- The straightforward high-ROI move is to batch pointwise work across many pixels instead of
  dispatching one matrix-vector multiply per pixel.
- A one-time image workspace allocation is acceptable if it enables that batched lowering without
  dispatch-time allocation.

## Specific Ideas

- Add a maintained batched pointwise path that packs pixel tiles into the kernel's preferred
  matrix shape and routes them through the existing f32 matmul kernel in larger chunks.
- Reuse or add one-time image scratch capacity so the new pointwise path does not allocate during
  dispatch.
- Preserve the existing HWC-facing caller contract so batch norm, skip paths, and later blocks do
  not need orchestration changes.
- Re-sample the image-only benchmark after the first lowering pass to decide whether a second
  in-phase optimization is warranted.

## Deferred Ideas

- Audio frontend optimization
- Text throughput work
- Liquid or `llama.cpp` reference-lane changes
- New image model scope or architecture changes
