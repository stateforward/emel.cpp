# Phase 59.1 Context

## Phase Boundary

Phase 59.1 reopens post-closeout work for one reason: the maintained steady-state ARM throughput is
still too low for an edge target, especially for image and audio embeddings. This phase keeps all
work on the maintained `src/emel/embeddings/generator` runtime and its maintained benchmark
surface, and it uses ROI-ranked profiling instead of broad cleanup.

## Implementation Decisions

### Scope
- Optimize only the maintained steady-state `embeddings/generator` path; helper-local or tool-only
  compute paths do not count.
- Throughput excludes `initialize(...)`; the benchmark must use an already-initialized generator
  instance.
- Keep the current stage split as the source of truth: `prepare`, `encode`, and `publish`.
- Use the existing Liquid reference lane only as an external comparison seam; missing Liquid assets
  are not a blocker for EMEL optimization work.

### Locked Benchmark Matrix
- Text comparison anchors: `Snowflake/snowflake-arctic-embed-s` and
  `ggml-org/embeddinggemma-300M-GGUF`.
- Vision comparison anchor: `LiquidAI/LFM2-VL-450M`.
- Audio comparison anchor: `fixie-ai/ultravox-v0_5-llama-3_2-1b`.
- Optional heavier vision/video ceiling if needed later: `Qwen/Qwen2.5-VL-3B-Instruct`.
- Treat those as comparison anchors and methodology targets, not as a reason to widen the current
  EMEL runtime scope beyond text, image, and audio embeddings.

### ROI Order
- Attack image `encode` first.
- Attack audio `prepare` second.
- Revisit text only after the image/audio passes, or when shared kernel/layout work materially
  closes the text gap for free.
- Reprofile after each landed change and stop pursuing any path that does not move the targeted
  stage.

### SML / Performance Constraints
- No hidden control flow in actions or detail helpers.
- No dequantize-to-f32 hot-path fallbacks or tool-only kernels.
- Any new timing or profiling hooks must remain observational only.
- Keep hot-path actions allocation-free and stay on the owning machine public event interface.

## Existing Code Insights

### Current Steady-State ARM Baseline
- Text: `~175.2 embeddings/s` (`5.709 ms`) with `prepare=0.0058 ms`, `encode=5.7033 ms`,
  `publish=0.00025 ms`
- Audio: `~18.0 embeddings/s` (`55.571 ms`) with `prepare=29.974 ms`, `encode=25.521 ms`,
  `publish=0.00067 ms`
- Image: `~1.34 embeddings/s` (`747.994 ms`) with `prepare=4.540 ms`, `encode=743.451 ms`,
  `publish=0.00079 ms`

### Current Hotspot Evidence
- Image time is still dominated by vision-tower compute, with remaining scalar
  `standard_conv_hwc` and `depthwise_conv_hwc` still material after the GEMM route cleanup.
- Audio time is still dominated by `prepare_audio_input`, which owns resample/STFT/mel/log
  frontend work.
- Text is materially healthier than image/audio, but it is still behind the strongest reported
  `llama.cpp` text-embedding numbers on comparable ARM hardware.

### Reference-Lane Reality
- The Liquid multimodal reference lane is maintained and buildable, but external reference
  throughput remains blocked on compatible multimodal assets.

## Specific Ideas

- Push the remaining image scalar spatial/depthwise conv work onto maintained ARM kernel paths or a
  more compute-friendly data layout.
- Remove any residual layout churn, fp16 decode, or buffer traffic still visible inside the
  image/audio towers.
- Precompute and reuse audio frontend state where it is invariant across requests, then vectorize
  the hottest loops in `prepare_audio_input`.
- Keep benchmark output centered on embeddings/s and stage timing for the real generator path.
- Record benchmark methodology in phase artifacts so later external comparisons against
  Arctic/Gemma/LFM2/Ultravox stay apples-to-apples.

## Deferred Ideas

- New modality scope beyond the current text/image/audio embeddings
- Python reference benchmarking
- Liquid asset bootstrap/downloading
- Video benchmarking unless it becomes necessary for a heavier vision ceiling
