# Phase 59.1 Summary

## Outcome

Phase 59.1 is complete. The maintained ARM throughput pass landed in the real
`src/emel/embeddings/generator` runtime, with ROI-ranked work on image `encode` and audio
`prepare`, and the maintained benchmark now shows materially better steady-state throughput for the
worst modality and a measurable image improvement without helper-local fallbacks.

## Delivered

- Reworked standard HWC conv paths to lower through the maintained f32 matmul kernel path using a
  scratch patch buffer instead of the previous scalar output-channel nest.
- Packed depthwise conv weights once at bind time into a kernel-major channel-contiguous layout and
  rewrote depthwise accumulation to run as a vectorizable channel loop.
- Replaced the audio frontend's naive per-bin DFT with a one-time-precomputed radix-2 FFT path.
- Added one-time audio FFT twiddle/bit-reversal tables and mel-bin nonzero ranges at bind time so
  the hot frontend avoids repeated trigonometric setup and dead mel-bin work.

## Measured Steady-State ARM Results

All numbers below exclude `initialize(...)` and come from isolated per-modality runs of the
maintained benchmark surface with `EMEL_BENCH_ITERS=10` and `EMEL_BENCH_RUNS=5`.

- Text: `~169.5 embeddings/s` (`5.901 ms`) with `prepare=0.0087 ms`, `encode=5.894 ms`,
  `publish=0.0006 ms`
- Audio: `~44.9 embeddings/s` (`22.269 ms`) with `prepare=0.159 ms`, `encode=22.108 ms`,
  `publish=0.0009 ms`
- Image: `~1.51 embeddings/s` (`661.395 ms`) with `prepare=4.547 ms`, `encode=656.891 ms`,
  `publish=0.0008 ms`

Compared with the pre-phase baseline:

- Image improved from `~1.37 embeddings/s` (`730.053 ms`) to `~1.51 embeddings/s`
- Audio improved from `~18.2 embeddings/s` (`54.821 ms`) to `~44.9 embeddings/s`

## Remaining Gap

- Audio is no longer frontend-bound in the same way; `prepare` collapsed from `~29.7 ms` to
  `~0.16 ms`, so the next audio bottleneck is now the encode tower.
- Image is still dominated by `encode`, so the next serious ARM pass should target the remaining
  batched pointwise/GEMM path in the vision tower.
- External `llama.cpp`/Liquid reference numbers still require a separate fresh comparison run with
  compatible assets; this phase only locked the methodology and improved the EMEL side.
