# Phase 46.1: Rename Planner-Family Wrapper Names To Remove Ambiguity And Wrapper Indirection Summary

**The planner family now uses the repo’s canonical naming again: the request path is
`plan_request` / `plan_scratch` / `plan_runtime`, the graph states use `state_*`, the planner and
child-mode callables use `effect_*`, the planner predicates use `guard_*`, and the planner surface
test locks the old wrapper-era names and temporary bare verb names out.**

## Performance

- **Duration:** 58 min
- **Completed:** 2026-04-06T02:37:39Z
- **Tasks:** 3
- **Files modified:** 22

## Accomplishments

- Renamed the top-level planner event/runtime surface in place and updated the direct maintained
  callers in generator, planner benches, and planner-focused tests.
- Replaced the top-level planner's wrapper-era state, guard, and action names with canonical
  `state_*`, `guard_*`, and `effect_*` naming while preserving behavior.
- Removed the remaining child-mode action trampolines by inlining the canonical
  `effect_begin_planning`, `effect_plan_simple_batches`, `effect_plan_sequential_batches`,
  `effect_plan_equal_batches`, `effect_plan_equal_primary_batches`, and `effect_reject_*` names
  into the mode action headers and rewiring the mode `sm.hpp` files and mode action tests to those
  names.
- Added source-level regression proof that explicitly fails if the old wrapper-era names reappear
  in the planner top-level surface or in the child-mode action headers.

## Decisions Made

- `complete_mode_request` stayed shared because duplicating the wrapper completion logic across
  three mode `sm.hpp` files would have added repetition instead of reducing indirection.
- SML transition rows in the mode machines still require callable objects, so the child-mode action
  surface now uses canonical `effect_*` named lambdas with inline bodies rather than trampoline
  lambdas around `_impl` helpers.
