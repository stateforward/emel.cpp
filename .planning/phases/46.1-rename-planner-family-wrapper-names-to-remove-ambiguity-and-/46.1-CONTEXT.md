# Phase 46.1 Context

## Phase Boundary

Phase 46.1 corrects the remaining naming-cutover miss inside the planner family. The user called
out that the current planner surface added better-named wrapper functions around existing
surfaces instead of renaming the ambiguous events, states, and functions directly.

The phase stays inside the planner family and its direct maintained consumers. It must reduce
ambiguity and wrapper indirection without widening scope into new behavior, generator redesign, or
benchmark work.

## Implementation Decisions

### Naming Closure
- Rename the planner's top-level trigger/runtime event types away from generic `request`,
  `request_ctx`, and `request_runtime` naming.
- Rename the top-level planner states to describe their role in the graph instead of generic
  wrapper-era names.
- Rename the top-level planner guard/action callables to direct planner names rather than
  `guard_*` and `effect_*` wrappers.

### Scope Discipline
- Keep behavior stable and stay inside `src/emel/batch/planner`, planner-focused tests, and direct
  maintained call sites that instantiate planner requests.
- Preserve additive public aliases like `emel::BatchPlanner`; this phase is about ambiguous
  planner-family wrapper names, not removing the approved top-level public type surface.
- Shared helper seams may stay if they remove duplication, but they should be named for their real
  responsibility rather than sounding like temporary wrappers.

### Proof Expectations
- Add a failing planner-family regression test first that expects the renamed direct surface.
- Re-run focused planner tests and the full quality gates after the rename lands.

## Existing Code Insights

### Current Ambiguity
- `src/emel/batch/planner/events.hpp` still exposes generic trigger/runtime names:
  `event::request`, `event::request_ctx`, and `event::request_runtime`.
- `src/emel/batch/planner/sm.hpp` still uses transient top-level state names such as
  `state_validate_decision`, `state_mode_decision`, and `state_publishing`.
- `src/emel/batch/planner/actions.hpp` and `src/emel/batch/planner/guards.hpp` still expose
  wrapper-style callables like `effect_begin_plan` and `guard_inputs_are_valid`.

### Direct Consumers
- Planner-focused tests under `tests/batch/planner/`
- Generator planning dispatch in `src/emel/generator/actions.hpp`
- Planner benchmark surfaces in `tools/bench/batch/planner_bench.cpp` and
  `tools/bench/generation_bench.cpp`

## Deferred Ideas

- Child-mode guard/action naming cleanup beyond what is required to adopt the renamed top-level
  planner events remains out of scope for this urgent closeout.
- Benchmark snapshot warning cleanup remains separate tech debt.
