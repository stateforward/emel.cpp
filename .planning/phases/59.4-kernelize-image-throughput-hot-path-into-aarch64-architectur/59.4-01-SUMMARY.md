# Phase 59.4 Summary

## Outcome

Phase 59.4 is complete. The maintained ARM image lane now calls kernel-owned pointwise and
depthwise entry points in `src/emel/kernel/aarch64/actions.hpp` instead of owning the long-lived
hot numeric loops in `src/emel/embeddings/generator/detail.hpp`. The maintained
`embeddings/generator` request path stayed unchanged, behavior remained verification-clean, and
the post-move profile shows the image hotspot is now explicitly owned by the kernel layer.

## Delivered

- Added explicit image-kernel request types and entry points:
  - `emel_image_pointwise_f32_request`
  - `emel_image_depthwise_f32_request`
  - `execute_neon_image_pointwise_f32<...>(...)`
  - `execute_neon_image_depthwise_f32(...)`
- Rewired `pointwise_conv_hwc_direct_f32_impl(...)` so the ARM hot path calls the kernel layer
  instead of running the fused pointwise loop in `detail.hpp`.
- Rewired `depthwise_conv_hwc(...)` so the ARM hot path calls the kernel layer instead of running
  the depthwise loop in `detail.hpp`.
- Added focused AArch64 parity tests for the new kernel-owned pointwise and depthwise entry
  points.

## Measured Maintained Image Result

All numbers below exclude `initialize(...)` and come from the maintained image-only benchmark
surface with `EMEL_BENCH_CASE_FILTER=image`, `EMEL_BENCH_ITERS=10`, and `EMEL_BENCH_RUNS=5`.

- After this phase: `96.713 ms/op` (`~10.34 embeddings/s`)
- Stage split:
  - `prepare=4.568 ms`
  - `encode=92.151 ms`
  - `publish=0.001 ms`
- Compared with the previous maintained image result (`~92.72 ms/op`, `~10.79 embeddings/s`),
  this phase is throughput-flat to slightly slower. The point of the phase was kernel ownership
  and contract cleanup, not a claimed speedup.

## Post-Move Ownership And ROI

The focused sample at `/tmp/emel_image_bench_59_4.sample.txt` shows the dominant image hotspot is
now owned by kernel symbols rather than `detail.hpp`:

- Kernel-owned pointwise work in `execute_neon_image_pointwise_f32<...>(...)` remains the clear
  dominant cost.
- Kernel-owned depthwise work in `execute_neon_image_depthwise_f32(...)` is the next
  image-specific hotspot.
- The next ROI remains pointwise kernel optimization inside `src/emel/kernel/aarch64`, not more
  ownership cleanup in `detail.hpp`.

## Notes

- This phase intentionally did not change the embedding-generator state machine. It only moved the
  proved numeric hot loops into the AArch64 kernel layer and republished maintained image
  evidence.
