# Phase 59.4 Context

## Phase Boundary

Phase 59.4 exists because the maintained single-thread image request path is still too slow for
the edge target, and the user explicitly does not want the long-lived hot numeric loops to remain
owned by `src/emel/embeddings/generator/detail.hpp`. This phase keeps the maintained TE image
request on the public `embeddings/generator` event path, but moves the dominant ARM image compute
into `src/emel/kernel/aarch64`.

## Implementation Decisions

### Scope
- Keep the source-of-truth benchmark on `tools/bench/embedding_generator_bench.cpp`.
- Keep the work steady-state only; `initialize(...)` remains excluded from throughput claims.
- Limit the phase to the maintained image lane.
- Move the image hot path into AArch64 kernel code without changing the embedding-generator state
  machine structure.

### Kernelization Target
- The first must-have kernel move is fused pointwise `1x1` f32 with optional batch norm and ReLU,
  because that is still the dominant image cost.
- The second kernel move is depthwise `3x3` same-padding stride `1/2` if it stays bounded and
  verification-clean after the pointwise move.
- One-time bind-time or reserve-time prepack is allowed; dispatch-time allocation is not.

### Constraints
- No hidden runtime behavior or outcome selection in `detail.hpp`.
- No helper-local benchmark shortcuts.
- No reference-lane compute, no tool-only runtime, and no public API widening.
- The image tower must remain on the maintained `process_event(...)` path.
- The end state on ARM should be architecture kernels in `src/emel/kernel/aarch64`, not long-term
  hot loops in `src/emel/embeddings/generator/detail.hpp`.

## Existing Code Insights

### Current Maintained Image Evidence
- Latest maintained image result recorded for the current image lane:
  - `~92.72 ms/embedding`
  - `~10.79 embeddings/s`
- Latest hotspot attribution from the maintained image-only sample:
  - `~79%` pointwise image kernel work
  - `~15%` depthwise image work
  - `~6%` shared GEMM substrate

### Current Ownership Problem
- The current image pointwise hot loop lives in
  `pointwise_conv_hwc_direct_f32_impl(...)` inside
  `src/emel/embeddings/generator/detail.hpp`.
- The current image depthwise loop lives in
  `depthwise_conv_hwc(...)` inside
  `src/emel/embeddings/generator/detail.hpp`.
- Those paths are now data-plane-proven, but they are still owned by the embedding generator layer
  rather than the AArch64 kernel layer.

### Practical Bias
- Reuse the already-proven operand formats and bind-time prepack choices from the current image
  runtime rather than inventing a new model contract.
- Move the proven AArch64 pointwise numeric loop into `src/emel/kernel/aarch64/actions.hpp`
  behind explicit request types or helper entry points.
- Only keep fallback/scalar glue in `detail.hpp`; the ARM hot path should call the kernel layer.

## Specific Ideas

- Introduce explicit AArch64 image-kernel request types for fused pointwise and depthwise work.
- Move the current pointwise direct kernel body into AArch64 kernel code and call it from the
  maintained image runtime.
- If bounded, add a kernel-owned depthwise entry point for the hot `3x3` same-pad image cases and
  rewire the maintained image blocks to that path.
- Add focused kernel tests so the new kernel layer is locked against the existing scalar/reference
  image behavior.

## Deferred Ideas

- Multi-threading
- Audio or text throughput work
- Reference-lane comparison changes
- State-machine structure changes
