# Phase 40 Plan 01

## Objective

Add truthful EMEL-owned `Q1_0_g128` loading and native execution for the maintained Bonsai slice on
the existing `qwen3` generator path.

## Tasks

1. Reproduce the missing-support failure with focused doctests for GGUF type `41`,
   `Q1_0_g128` kernel execution, and model/runtime validation.
2. Move EMEL-owned packed/prepared pseudo-dtypes out of the upstream GGML/GGUF id range so loader
   and runtime code can represent real `Q1_0_g128` tensors without ambiguity.
3. Port the Prism scalar `Q1_0_g128` block layout, row dequant helper, and `Q1_0_g128 x Q8_0`
   dot-product path into `src/emel/kernel/detail.hpp`, then wire native dtype accounting and model
   quantized-path validation around it.
4. Verify that the shipped generator backend accepts the real Bonsai tensor dtype through the
   existing `qwen3` lane and keeps execution on the native quantized kernel path.

## Verification

- `cmake --build build/zig --parallel --target emel_tests_bin`
- `./build/zig/emel_tests_bin --test-case='*q1_0_g128*' --no-breaks`
- `./build/zig/emel_tests_bin --test-case='*qwen3*' --no-breaks`
- `./build/zig/emel_tests_bin --test-case='*generator_detail_*' --no-breaks`
- `./scripts/quality_gates.sh`

## Done Condition

EMEL loads Bonsai's real GGUF type `41` tensors as `Q1_0_g128`, model audits treat that dtype as
native quantized, and the shipped generator path can execute the maintained Bonsai slice without
whole-tensor dequantize-to-f32 fallback.
