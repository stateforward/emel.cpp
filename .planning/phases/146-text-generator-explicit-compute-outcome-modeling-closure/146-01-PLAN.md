# Plan 146-01: Explicit Compute Outcome Modeling Closure

## Goal

Close the remaining v1.17 audit blockers by moving graph compute route preconditions and
invalid/backend outcome decisions out of action-called text generator detail run-kernel wrappers and
into explicit parent/prefill actor guards and transition rows.

<rule_constraints>
## Rule Traceability

- `docs/rules/sml.rules.md` and AGENTS.md require runtime behavior choice to be expressed as
  guards and explicit transitions.
- Actions must execute bounded work for an already-selected path; they must not validate and route
  success/error outcomes through helper-local branching.
- Detail helpers called from actions must not choose what happens next through return values,
  `err_out`, selected callbacks, fallback selection, or route probing.
- Public parity, benchmark, and closeout claims must be traced through maintained source paths.
</rule_constraints>

## Tasks

1. Add source-backed regressions that fail when audited run-kernel wrappers branch on request plan,
   step kind, backend readiness, selected output pointers, chunk readiness, or `err_out`.
2. Add generator parent guards for decode graph compute preconditions and wire invalid/backend
   destinations before compute dispatch actions.
3. Add prefill actor guards for prefill graph compute preconditions and wire invalid/backend
   destinations before compute dispatch actions.
4. Simplify the audited `detail.hpp` run-kernel wrappers so they only run the already-selected
   numeric kernel and set no validation/outcome `err_out`.
5. Update tests that previously relied on malformed direct detail calls to verify actor-level
   behavior and source ownership instead.
6. Run focused generator tests, domain boundary checks, parity/benchmark evidence, and the scoped
   quality gate.
7. Record validation evidence and update roadmap, requirements, and state truthfully.

## Success Criteria

- Parent/prefill SML tables contain explicit guard-owned transitions for compute readiness failures.
- Action-called run-kernel callbacks do not inspect request plan kind, backend validity, bound state,
  output pointer availability, or chunk readiness to decide the success/error path.
- Source regressions fail if the Phase 146 violations return.
- Maintained generator behavior and public parity/benchmark entrypoints remain source-backed.
- Phase 146 artifacts include exact validation commands and results.
