# Phase 40.1 Context

## Phase Boundary

Close the remaining maintained Bonsai generation gap by adding an explicit Qwen-family RoPE domain
to the shipped EMEL runtime, including rope type, pair layout, and YaRN scaling semantics, without
adding any Bonsai-only behavior lane.

## Implementation Decisions

### Domain Shape
- Qwen-family rotary behavior is modeled as explicit runtime metadata: rope type plus rope scaling
  type and the small set of YaRN parameters required by the maintained slice.
- Unsupported rotary domains fail explicitly during generator prepare as `model_invalid` instead of
  silently falling back to plain adjacent-pair RoPE.
- The shipped generator keeps one existing `qwen3` execution lane; the widening is a Qwen-family
  domain contract, not a new architecture family or a Bonsai-specific branch.

### Metadata Truth
- GGUF metadata loading must publish rope type and scaling truth into `model::data` so runtime
  behavior is derived from loaded model data instead of tool-local parsing.
- `qwen3` defaults to the `neox` pair layout, while YaRN metadata such as original context length,
  scaling factor, and beta values flow through the same shared model-metadata contract.
- Bench and parity tooling reuse the same metadata-population helper so EMEL and reference lanes
  stay honest about the loaded rope domain.

### Parity Honesty
- The remaining Bonsai mismatch was at the first generated-token Q/K boundary after prompt text and
  prompt tokens already matched, which isolates the missing behavior to runtime RoPE semantics.
- Once the EMEL-owned rope domain became truthful, the maintained Qwen generation fixture needed
  to move from a stale append-only snapshot to live reference generation so the prior Qwen anchor
  remains a real maintained comparator instead of a frozen historical artifact.

## Existing Code Insights

### Reusable Assets
- `src/emel/generator/detail.hpp` already owned the shipped RoPE application path and was the
  correct place for the bounded numeric helpers and explicit runtime-domain validation.
- `src/emel/model/detail.hpp` already centralized GGUF metadata extraction and could be widened to
  publish rope-domain truth without changing machine topology.
- `tools/paritychecker/parity_runner.cpp` and `tools/bench/generation_bench.cpp` already build
  fully isolated EMEL and reference lanes, so they could consume the shared rope metadata helper
  without violating lane separation.

### Integration Points
- `src/emel/model/data.hpp` is the canonical place to model rope-domain enums as loaded model
  truth.
- `src/emel/model/detail.hpp` is the correct metadata seam for rope-domain extraction from GGUF.
- `src/emel/generator/detail.hpp` is the shipped runtime seam for rope-domain validation and
  Qwen-family RoPE application.
- `tools/generation_fixture_registry.hpp` is the maintained-fixture contract seam for keeping the
  Qwen anchor honest once live reference parity is trustworthy again.
