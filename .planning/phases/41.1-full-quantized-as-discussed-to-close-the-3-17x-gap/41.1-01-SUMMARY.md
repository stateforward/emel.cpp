# Phase 41.1 Summary

Phase 41.1 closed the biggest remaining Bonsai performance gap without widening the runtime
surface or adding a Bonsai-only path. The shipped `Q1_0_g128 x q8_0` operand path now uses an
EMEL-owned AArch64 NEON/dotprod implementation in `src/emel`, and the quantized-path audit no
longer treats quantized token embedding as an approved dense fallback.

The key runtime change was operand-path truth, not benchmark-only scaffolding. `src/emel` now
consumes quantized Bonsai token-embedding rows as a native stage and the maintained `Q1_0_g128`
decode path no longer relies on the previous scalar dot-product loop on supported AArch64 hosts.
That keeps the phase aligned with the repo contract: no tool-local compute fallback, no
whole-tensor dense substitute, and no model-specific hidden branch.

The measured implementation result moved materially. On the isolated maintained Bonsai smoke lane,
the EMEL-vs-Prism ratio improved from `3.173x` slower to `1.343x` slower while live parity stayed
green and `./scripts/quality_gates.sh` returned `0`. That closes the performance-closure phase and
leaves benchmark publication as the remaining milestone step.
