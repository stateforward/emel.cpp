# Phase 41.1 Context

## Phase Boundary

Close the remaining maintained Bonsai performance gap before benchmark publication by removing the
largest remaining non-native `Q1_0_g128` seams on the shipped generator path. This phase is about
runtime operand-path closure in `src/emel`, not about widening model scope, changing benchmark
policy, or adding Bonsai-only behavior.

## Implementation Decisions

### Locked Runtime Shape
- Bonsai remains an existing `qwen3` slice; no new `bonsai` or `prismml` runtime family is
  allowed.
- Runtime control flow must stay explicit through guards, states, and transitions. Numeric work
  belongs in `detail.hpp`; there must be no hidden action branching.
- The reference lane stays confined to tooling. Phase 41.1 must port arithmetic and operand-path
  behavior into `src/`, not depend on `tools/bench` or `tools/paritychecker` for shipped runtime
  behavior.

### Performance Gap To Close
- The focused implementation bench currently reports Bonsai at `3.173x` slower than the Prism
  lane on `generation/preloaded_request/bonsai_1_7b_prompt_hello_max_tokens_1`.
- The parity/runtime audit still reports `token_embedding` as
  `approved_dense_f32_by_contract`, not `native_quantized`.
- `src/emel/kernel/detail.hpp` still executes `Q1_0_g128 x q8_0` matmul and argmax through scalar
  loops, while the Prism fork already carries an ARM NEON/dotprod implementation for the same
  operand class.

### Chosen Phase Shape
- Treat the remaining gap as an operand-domain problem, not a model exception.
- Promote quantized token embedding handling to an explicit native stage when the stored tensor is
  quantized and the shipped runtime consumes it directly without a whole-tensor dense fallback.
- Port the Prism/ggml ARM `Q1_0_g128 x q8_0` vector-dot behavior into the EMEL-owned AArch64
  kernel domain using the existing explicit guard/action dispatch style.
- Keep phase-local proof narrow: improve the maintained Bonsai implementation bench materially,
  preserve parity, and keep the full quality gate green.

### Guardrails
- No new machine is needed unless runtime orchestration structure itself changes, which this phase
  does not currently require.
- Benchmark snapshots and docs publication remain Phase 42 work. This phase should use
  implementation-mode bench evidence and preserve the existing snapshot policy.

## Specific Ideas

- Add explicit AArch64 SIMD guards and actions for `Q1_0_g128` vector matmul and argmax on the
  shipped kernel machine.
- Replace the remaining scalar `Q1_0_g128` hot decode path with the ported ARM vector-dot logic
  from the Prism reference lane.
- Tighten the quantized-path audit so token embedding becomes `native_quantized` only when the
  shipped runtime truly consumes the quantized row path directly.
- Re-run the isolated Bonsai implementation bench after the runtime change and use the measured
  ratio as the phase acceptance signal.

## Deferred Ideas

- Broader multi-model `Q1_0_g128` publication remains Phase 42 or later.
- Any non-ARM backend specialization remains out of scope for this milestone.
