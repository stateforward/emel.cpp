# Phase 25.1: Restore Canonical Flash-Attention Dispatch On The Maintained Generator Path - Research

**Researched:** 2026-03-25
**Domain:** Canonical generator flash-dispatch regression
**Confidence:** HIGH

## Summary

The current regression is not a missing kernel implementation. The shipped generator runtime still
has `make_flash_attn_request(...)` and `dispatch_flash_attention(...)`, but `run_layer(...)` now
calls the manual `compute_attention(...)` helper on the supported canonical path. That drift is
visible all the way up the maintained proof surfaces: generator lifecycle tests assert zero flash
dispatch on the canonical fixture, paritychecker tests assert zero flash dispatch in generation
proof output, and the stored benchmark/docs artifacts now publish `flash_dispatch_calls=0` for the
maintained compare case.

The intended contract is still documented in the archived v1.3 milestone artifacts. Phase 15
established that maintained generation proof should distinguish optimized versus shared flash
dispatch, and Phase 16 established that the canonical compare workload on AArch64 should publish
`optimized_flash_dispatch_calls=2` and `shared_flash_dispatch_calls=0`. Phase 25.1 therefore
should restore the existing flash dispatch seam on the supported shipped path, keep the
`flash_kv_width_mismatch` fixture as the explicit nonflash negative path, and then realign
generator/parity/bench proof with that contract.

## Recommended Plan Split

### 25.1-01: Restore Runtime Flash Selection And Regression Tests

- Add failing regression coverage that proves the canonical fixture should dispatch flash again,
  while preserving the explicit nonflash mismatch fixture.
- Change the shipped generator runtime to select `dispatch_flash_attention(...)` for supported
  requests and keep `compute_attention(...)` only for the unsupported explicit fallback path.
- Verify with focused generator tests and paritychecker tests before any artifact refresh.

### 25.1-02: Re-Align Maintained Proof And Publication

- Update paritychecker tests and compare-mode validation so the maintained proof surfaces reject
  zero-flash attribution on the supported canonical path.
- Re-run focused compare/parity surfaces and, if the runtime fix changes stored benchmark/docs
  artifacts, stop at the normal snapshot approval boundary unless user approval is already clear for
  that publication refresh.
- Close with `scripts/quality_gates.sh` and restore the generated timing snapshot afterward.

## Risks

- The earlier long-decode parity repair intentionally restored masked nonflash attention semantics;
  Phase 25.1 must preserve current `1/10/100/1000` parity while restoring supported flash usage.
- The canonical flash path must remain host-aware: AArch64 should use optimized flash with no
  shared fallback, while other hosts should stay explicit and truthful.
- Stored benchmark/docs artifacts will become stale once flash attribution is restored, so
  publication steps must follow the repo's explicit approval rule.
