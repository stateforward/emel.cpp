# Phase 58.1 Context

## Phase Boundary

Phase 58.1 is an urgent follow-on to the maintained embedding-generator benchmark. The benchmark is
now real and steady-state, but the first ARM numbers showed that image and audio throughput are far
below an acceptable edge-runtime target. This phase closes the immediate performance gap inside the
maintained `src/emel/embeddings/generator` path rather than in ad hoc helper code.

## Implementation Decisions

### Scope
- Keep the work on the maintained embedding generator runtime and its maintained benchmark surface.
- Treat initialize cost as a separate one-time concern; steady-state throughput excludes
  `initialize(...)`.
- Use the inserted phase to land the ARM-path cleanup already identified from profiling, then add
  auditable phase timing so future work can target the real bottlenecks precisely.

### Locked Performance Directions
- Route steady-state f32 and q8 hot matmul/pointwise-conv work through the maintained ARM kernel
  paths instead of local scalar helpers where the maintained kernels already cover the operand
  shape.
- Stop decoding fp16 inside inner loops; bind or expand once into a compute-friendly layout during
  non-dispatch setup.
- Remove full-buffer memcpy ping-pong in the image and audio towers when alternating buffers is
  sufficient.
- Split the maintained benchmark into `prepare`, `encode`, and `publish` timing so the profile can
  distinguish tokenizer/frontend cost from tower compute and final output publication.

### SML / Control-Flow Constraints
- The benchmark timing split must still be modeled through the embedding generator state machine.
- Do not reintroduce hidden runtime routing in actions or helper-local callback dispatch.
- Any optional timing surface must be observational only; it must not choose what runs next.

## Existing Code Insights

### Already Landed In The Worktree
- `src/emel/embeddings/generator/detail.hpp` already routes steady-state `matmul_f32` and
  `matmul_q8_0` through the maintained ARM kernel helpers on AArch64.
- fp16 matrix and conv weights are already expanded once at bind time in the embedding-generator
  runtime.
- The image and audio towers already swapped away the worst full-buffer copy-backs by carrying
  current/alternate buffers through block execution.

### Remaining Gap
- The maintained benchmark still only reports whole-request `ns_per_op`.
- There is no maintained `prepare` / `encode` / `publish` timing surface exposed through the real
  generator machine after the earlier callback-style prototype was removed for hidden control flow.
- Future optimization work still needs per-stage evidence, especially for the audio frontend and
  the remaining scalar image/audio conv paths.

## Specific Ideas

- Add an observational benchmark timing payload to the public embedding request events so the
  maintained benchmark can retrieve per-stage timings without calling helper-local functions
  directly.
- Stamp phase boundaries inside the existing prepare, encode, and publish actions only; keep all
  routing decisions in guards and `sm.hpp`.
- Extend the dedicated embedding benchmark runner to record and print median `prepare_ns`,
  `encode_ns`, and `publish_ns` alongside total `ns_per_op` for text, image, and audio cases.

## Deferred Ideas

- New native ARM kernels for the remaining scalar spatial/depthwise conv paths
- A rewritten audio FFT/mel frontend
- Additional benchmark cases beyond the maintained three-modality steady-state surface
